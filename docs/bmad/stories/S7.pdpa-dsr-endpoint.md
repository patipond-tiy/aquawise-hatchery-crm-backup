# Story S7: PDPA Data Subject Request (DSR) Endpoint

Status: done

Severity: P0 — required for launch (PDPA compliance, response window 30 days)

## Story

As an end user (or someone acting on behalf of one) under Thailand's PDPA,
I want to request a copy of my personal data and request its deletion,
so that the company complies with PDPA Section 30 (right to access) and Section 33 (right to erasure) without manual SQL.

## Acceptance Criteria

1. Authenticated `GET /api/dsr/export` endpoint returns a JSON download of all personal data tied to the calling user's auth UID across hatchery-crm tables
2. Authenticated `POST /api/dsr/delete` endpoint anonymizes (not hard-deletes) personal data while preserving financial records (Revenue Department requires 7-year retention)
3. Both endpoints conform to the shared DSR spec at `docs/aquawise-updated-docs/DSR-SPEC.md` (the in-repo mirror; source of truth is `aquawise-ecosystem/aquawise-docs/DSR-SPEC.md`; line-bot implements the same contract from its own mirror)
4. Every DSR request writes an `audit_log` entry with action `dsr.export` or `dsr.delete`
5. Anonymization replaces these fields: `customers.name` → `redacted_<uuid>`, `customers.phone` → `null`, `customers.address` → `null`, `customer_line_ids.line_user_id` → `null` (delete row), `audit_log.actor_user_id` → kept (legal requirement) but joined-displayed as `redacted`
6. The "financial preservation" rule: invoice rows and `subscription_events` rows are kept intact but the linked customer name shows `redacted_<id>` in any future export
7. Rate-limit: 5 requests per user per 24h to prevent abuse
8. Click-through: a test user can request export, receive JSON; request delete, confirm their name no longer appears in any UI read

## Dev Notes

### Architecture Constraints
- These are API routes, not server actions — they need explicit auth checks (no automatic Origin/Host CSRF protection from Next.js)
- Use the service-role client ONLY for the SECURITY DEFINER RPC call; everything else goes through the user-session client
- The endpoint MUST work for a user with no remaining membership (post-revocation) — they still own their PII

### RBAC
- DSR is the user's own request — no `can()` check; the trust boundary is `getUser()` returning a real user. Any user can DSR themselves regardless of role.

### RLS
- The SECURITY DEFINER RPC runs with elevated privileges. Audit it carefully — it's a privileged write.

### Brand voice
- The Delete confirmation copy is a hard moment for a customer. Voice: `ลูกหลานที่เรียนมา` — respectful, factual, no exclamation, no "Are you sure?!" Use formal Thai.
