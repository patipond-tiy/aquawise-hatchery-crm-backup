# Story D2: Send a Quote in One Tap

Status: draft

## Story

As a Rep,
I want to send a co-branded quote (price per size, payment terms, lead time) to the farm via LINE in one click,
so that I get the lot ordered before a competitor calls.

## Acceptance Criteria

1. "เสนอราคา" opens the `quote` modal pre-filled with the customer's last package, current `prices` rows, and the hatchery's lead time
2. Rep can edit line items and add a note before sending
3. Submit inserts a `quotes` row (status `sent`) and enqueues a `quote` Flex template to `line_outbound_events`
4. Farmer receives an informational quote Flex Message (line items, price, lead time, validity). The "ตอบรับ" button deep-links to the hatchery's existing LINE OA so the farmer replies through the relationship channel the operator already runs — **no LIFF chat thread, no in-app chat surface** (two-way LIFF is Phase H3 per `prd.md` §6 row G; CEO-ratified deferral 2026-05-15, see `../decisions-2026-05-15-fix-review.md` §D2)
5. Quote status is **rep-driven in H1**: the rep sets `accepted | declined | expired` from the CRM when the farmer responds through the existing phone/LINE-OA channel. The status machine is `sent → accepted | declined | expired`; H1 has no automated farmer-side status callback (that arrives with H3 two-way)
6. Quotes are visible on the customer detail page under a "Quotes" tab

## Dev Notes

> ⚠ **Hypothesis dependency.** The one-tap branded-quote workflow assumes the hatchery↔customer relationship framing in `docs/aquawise-updated-docs/06-aquawise-hatchery-customer-doc.md` §3 Job 3, which is a **v0.5 hypothesis pending P'Bunjong validation** — not confirmed customer reality. Treat the quote-send flow as a scaffold to validate, not a settled requirement. The send-only scope (no two-way chat) is CEO-ratified 2026-05-15 (`../decisions-2026-05-15-fix-review.md` §D2): the relationship work stays the operator's own job via their existing LINE OA / phone, per nursery customer doc §7 ("we tell him who to send to") and §10 ("you are not building a CRM").

### Architecture Constraints
- All API calls via `lib/api/index.ts` facade — never Supabase directly from pages
- Server actions at `app/[locale]/(dashboard)/restock/actions.ts`; any action that writes must call `requireActiveSubscription()` first
- `'use client'` opt-in only; server components by default
- No `any` in app code; `as` casts allowed only in adapters (e.g., `rowToQuote` adapter in `lib/api/supabase.ts`)

### Blockers
This story is **BLOCKED** by:
- **G3'.i** (Send-only Flex messaging worker): the `line_outbound_events` queue exists but nothing processes it. Enqueuing the `quote` Flex event is safe to implement now; the farmer will not receive it until the bot worker is deployed.
- **G1.i** (LINE binding): quotes can be inserted and displayed in the CRM without a bound `line_id`, but the Flex Message cannot be delivered until `customers.line_id` is populated.

The CRM-side implementation (Tasks 1–6) can proceed without the LINE worker being live. The task table should note G3'.i and G1.i as runtime prerequisites for the `.v` live verification.
