# Story S3: File Upload ‚Äî Validate Magic Bytes Server-Side

Status: ready-for-dev

Severity: P1 ‚Äî pre-launch hardening

## Story

As an Owner,
I want logo uploads validated by their actual file bytes (not just the client-supplied Content-Type),
so that an attacker cannot upload HTML or polyglot files disguised as images and have them served from our public Storage bucket.

## Acceptance Criteria

1. `lib/supabase/storage.ts:uploadHatcheryLogo` reads the first 8 bytes of the upload and rejects anything that is not a known JPEG / PNG / WebP / GIF signature
2. The filename extension is sanitized to a whitelist (`jpg`, `jpeg`, `png`, `webp`, `gif`) ‚Äî path-traversal characters (`/`, `..`, control chars) are stripped
3. File size is validated in app code (max 2 MB) **before** the upload begins
4. The `Content-Type` passed to Supabase Storage matches the detected magic-byte type ‚Äî never the client-asserted type
5. A test in `tests/storage/upload.test.ts` covers: valid PNG accepts, HTML-as-PNG rejects, path-traversal filename rejects, oversized file rejects
6. The settings page upload UI surfaces a friendly Thai/English error for each rejection class

## Tasks / Subtasks

- [ ] Task 1 ‚Äî Magic-byte validation helper (AC: #1, #4)
  - [ ] Create `lib/supabase/file-validation.ts` with `detectImageType(file: File): Promise<'jpeg' | 'png' | 'webp' | 'gif' | null>`
  - [ ] Read first 12 bytes via `file.slice(0, 12).arrayBuffer()`
  - [ ] Compare prefix to known signatures (see `security.md` ¬ß1 for byte sequences)
  - [ ] Return `null` if no match
- [ ] Task 2 ‚Äî Filename sanitization (AC: #2)
  - [ ] In `lib/supabase/storage.ts`, replace `file.name.split('.').pop()` with: `file.name.replace(/[^a-zA-Z0-9.]/g, '').split('.').pop()?.toLowerCase()`
  - [ ] Reject if the resulting extension is not in `['jpg','jpeg','png','webp','gif']`
- [ ] Task 3 ‚Äî Size cap (AC: #3)
  - [ ] At the top of `uploadHatcheryLogo`, `if (file.size > 2_097_152) return { ok: false, error: 'File too large' }`
- [ ] Task 4 ‚Äî Wire validation into upload flow (AC: #1, #4)
  - [ ] Call `detectImageType(file)` before `supabase.storage.from(BUCKET).upload(...)`
  - [ ] If `null`, return `{ ok: false, error: 'Invalid image' }`
  - [ ] Pass `contentType` derived from the detection result, NOT from `file.type`
- [ ] Task 5 ‚Äî Tests (AC: #5)
  - [ ] `tests/storage/upload.test.ts` ‚Äî 4 cases:
    - valid PNG (header `89 50 4E 47`) ‚Üí accepts
    - HTML body with `Content-Type: image/png` (header `3C 21 44 4F` for `<!DO`) ‚Üí rejects with "Invalid image"
    - filename `../../etc/logo.png` ‚Üí rejects with "Bad filename"
    - 3 MB file ‚Üí rejects with "File too large"
  - [ ] Use `vi.mock('@/lib/supabase/server')` matching the existing pattern in `tests/api/`
- [ ] Task 6 ‚Äî i18n + UI error surfacing (AC: #6)
  - [ ] Add keys: `Settings.profile.upload.errors.invalid_image`, `bad_filename`, `too_large`
  - [ ] Map server-side error strings to keys in `app/[locale]/(dashboard)/settings/page.tsx` upload handler
- [ ] Task 7 ‚Äî Manual verification
  - [ ] Try to upload a `.html` file renamed to `.png` ‚Üí rejected with friendly message
  - [ ] Upload a real PNG ‚Üí accepted and visible on the public scorecard URL

## Dev Notes

### Architecture Constraints
- This is a `'server-only'` module ‚Äî `lib/supabase/storage.ts` already imports `'server-only'`
- Validation runs server-side because client-side checks can be bypassed
- Magic-byte detection is fast (12-byte read) ‚Äî fine to do synchronously before the upload call
