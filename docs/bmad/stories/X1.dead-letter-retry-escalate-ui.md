# Story X1: Dead-Letter Retry / Escalate UI

Status: done

## Story

As an Owner,
I want to see a list of LINE pushes that failed all retries and choose to retry, edit, or resolve them,
so that failed messages to high-value farmers do not silently disappear.

## Acceptance Criteria

1. New page at `app/[locale]/(dashboard)/settings/messaging-failures/page.tsx` lists all `line_outbound_events WHERE status='dead'` for the owner's nursery (RLS-scoped — no cross-tenant leakage)
2. Each row shows: customer name, template name, payload preview (first 120 chars, truncated), last error message, attempt count, first-failure timestamp (ICT, formatted as Thai date)
3. Per-row action — **Retry**: resets `status → pending`, increments `attempts`, inserts an `audit_log` row (`action='dead_letter_retry'`), triggers bot worker re-processing
4. Per-row action — **Edit & Retry**: opens an inline or modal editor for the `payload` JSON field; on save, updates `payload`, resets `status → pending`, inserts audit log row (`action='dead_letter_edit_retry'`)
5. Per-row action — **Mark Resolved**: sets `status → resolved` (or equivalent closed state), inserts audit log row (`action='dead_letter_resolved'`), removes the row from the active list
6. Bulk action: checkbox select multiple rows → "Retry selected" button resets all selected events to `pending` in a single server action; audit log rows inserted for each
7. Once-daily digest: if any `line_outbound_events` row is `dead` for the nursery, a digest notification fires to the relevant channel ("5 ข้อความรอการแก้ไข") — entry added to `vercel.json` cron and handled in `app/api/cron/daily/route.ts`
8. Page is accessible to `owner` role only — guarded by the `ops:view` RBAC action; other roles receive a 403

## Dev Notes

### Architecture Constraints
- All API calls via `lib/api/index.ts` facade — never Supabase directly from pages
- Server actions at `app/[locale]/(dashboard)/settings/messaging-failures/actions.ts`
- `'use client'` opt-in only; the page itself can be a server component; the row-level action buttons must be a client component (they need `useTransition` for optimistic feedback)
- No `any` in app code
- Audit log INSERT uses the Supabase server client (not service-role) — the RLS policy on `audit_log` should allow nursery-member inserts; if using service-role in server actions, that is also acceptable

### `resolved` Status
`line_outbound_events.status` currently has values `pending | sending | sent | failed | dead`. The "Mark Resolved" action needs a `resolved` variant. If the DB enum does not include it, add an incremental migration:

```sql
ALTER TYPE line_event_status ADD VALUE IF NOT EXISTS 'resolved';
```

Or alternatively, set `status='dead'` and add a separate `resolved_at timestamptz` column. Decide before writing the migration — keep it consistent with the existing enum pattern in `006_line_integration.sql`.

- Page and actions require owner-only access. The guard uses `ops:view`, which is semantically correct — it means "can view operational internals" and is distinct from `billing:manage` ("manage Stripe subscription").
- **`ops:view` does NOT currently exist in `lib/rbac.ts`** (confirmed: `lib/rbac.ts:5-15` lists no `ops:view` entry). Any code calling `can(role, 'ops:view')` will produce a TypeScript compile error until Task 1 adds it. Task 1 is a hard prerequisite — do not write any `ops:view` guards before the Action union entry exists.
- After Task 1: `RULES['ops:view'] = ['owner']`

- `counter_staff`, `lab_tech`, and `auditor` do NOT have `ops:view`
