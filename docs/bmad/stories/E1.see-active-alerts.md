# Story E1: See Active Alerts

Status: ready-for-dev

## Story

As an Owner,
I want to see all open alerts across batches, ranked by severity,
so that I respond before reputation damage.

## Acceptance Criteria

1. `/alerts` shows all `closed=false` alerts, sorted by `sev DESC, date DESC`
2. Severity stat cards display high / medium / low counts
3. Each alert shows: severity icon, batch link, affected farms, recommended action

## Dev Notes

### Architecture Constraints
- All API calls via `lib/api/index.ts` facade — never Supabase directly from pages
- Server components by default; `'use client'` opt-in only
- No `any` in app code; `as` casts allowed only in adapters (`lib/api/supabase.ts`)
- RBAC: any authenticated member can read alerts (`customer:read` gate applies broadly); check `lib/rbac.ts` before adding role restrictions

### Files to Touch
The implementation (`.i`) is complete (commit `ddf6ea3`). The following files should be read during verification — no code changes expected unless a gap is discovered.

| File | Role |
|------|------|
| `lib/api/supabase.ts` (lines 162–189) | `listAlerts()` — severity sort fix lives here |
| `lib/mock/api.ts` | Mock `listAlerts()` — verify it also returns `closed=false` only |
| `app/[locale]/(dashboard)/alerts/page.tsx` | Page that calls `listAlerts()` and renders cards + stat counts |
| `tests/api/list-alerts.test.ts` | Regression test asserting the `.order()` chain |

### API Facade Methods
`listAlerts()` is the sole facade method for this story. Signature (live path, `lib/api/supabase.ts`):
```ts
listAlerts(): Promise<Alert[]>
// Returns: alerts where closed = false, ordered sev DESC, created_at DESC
```
No new facade methods needed.

### RBAC
- `alert:close` is the write action; read access is not further restricted beyond nursery membership
- Do not add role-gating to the read path unless the spec explicitly requires it
