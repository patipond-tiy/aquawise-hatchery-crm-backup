# Story K4: Batch-Warning Webhook Publisher (CRM → LINE bot)

Status: done

> **Contract:** conforms to `docs/aquawise-updated-docs/K-INTEGRATION-CONTRACT.md`. Webhook JWT = §3 (`iss=nursery-crm, aud=line-bot-webhook` — MUST differ from the read-side audience); payload + severity routing + retry = §7. Where this story and the contract disagree, the contract wins.

## Story

As an Owner or Manager,
I want a warning I publish on a batch to be delivered to every farmer whose pond is feeding that batch via LINE,
so that downstream farmers can react before pond damage compounds (LINE bot Story K.12).

## Acceptance Criteria

1. **Publish surface:** From the batch detail page (`/[locale]/(dashboard)/batches/[id]`), a "แจ้งเตือนล็อตนี้" button opens a modal collecting `severity` (`info`/`warning`/`critical`), `title_th`, `body_th`, and `action_th`. RBAC: only `owner` and `counter_staff` may publish (uses `can(role, 'alert:close')` action — the same action gates disease-alert closure and notify in Epic E).
2. **Server action `publishBatchWarning(batchId, severity, title_th, body_th, action_th)`** at `app/[locale]/(dashboard)/batches/actions.ts`:
   - Inserts a `crm_event_log` row (`event_type='batch_warning'`, the payload above, fresh `correlation_id` UUID, `posted_at=now()`, `delivered_at=null`, `attempts=0`).
   - Returns the new `correlation_id` to the caller.
3. **Outbound delivery:** A new server-side function `deliverBatchWarning(correlationId)` POSTs the payload to `${LINE_BOT_BASE_URL}/api/crm-events/batch-warning` with body:
   ```jsonc
   {
     "batch_code": "B-A4F2K7",
     "severity": "critical",
     "title_th": "ตรวจพบ EHP",
     "body_th": "...",
     "action_th": "...",
     "posted_at": "<iso>",
     "correlation_id": "<uuid>"
   }
   ```
4. **Auth on the outbound side:** Sign an ES256 JWT with `iss=nursery-crm`, `aud=line-bot-webhook` (note: different audience than the read-side JWT in K2 to prevent confused-deputy attacks), `exp` 5 minutes. Sign with `CRM_WEBHOOK_JWT_PRIVATE_KEY` env var. The LINE bot verifies with the matching public key.
5. **Delivery is async + retried:**
   - For `severity='critical'`: invoke `deliverBatchWarning()` inline from the server action (caller awaits up to a 3-second timeout); if it 2xxs, set `delivered_at`; if not, fall through to the retry path.
   - For `severity='warning'`: enqueue only; the retry cron handles delivery within the next 5-min tick.
   - For `severity='info'`: enqueue only; the retry cron delivers as a regular tick.
   - Retry cron `app/api/cron/deliver-crm-events/route.ts` runs every 5 minutes; selects rows where `delivered_at IS NULL AND attempts < 12`; exponential back-off `min(2^attempts, 60)` seconds between attempts; after 12 attempts the row is left at `attempts=12, last_error=<final>` and surfaces in a dead-letter UI (deferred — track as `K4.t.deadletter` follow-up).
6. **Cron auth:** `Authorization: Bearer ${CRON_SECRET}` — `CRON_SECRET` is a **new** env var introduced by this story (added to `.env.example` in Task 5). There is no existing cron endpoint in the repo (`app/api/cron/` does not exist yet). The Stripe webhook at `app/api/webhooks/stripe/route.ts` uses `STRIPE_WEBHOOK_SECRET` verified via Stripe's HMAC signature (`stripe.webhooks.constructEvent`) — not a Bearer token — so `CRON_SECRET` is unrelated and does not "match" that convention. The Bearer-token pattern here follows the same simple shared-secret approach we will use for all future internal cron endpoints.
7. **Toast + audit:** On `publishBatchWarning()` success, show a Thai toast `"แจ้งเตือนล็อต {code} ส่งแล้ว — ฟาร์มที่ใช้ล็อตนี้จะได้รับการแจ้งเตือน"` and append to `audit_log` `event='batch_warning_published'`.
8. **Auto-publish from disease alerts (Story E4 mod):** When a disease alert is created with `severity='critical'` AND the alert is linked to one or more batches (`alert_batches` join — see Story E4 mod), the alert-creation server action automatically calls `publishBatchWarning()` for each linked batch using the alert's title and body. This auto-publish does NOT skip the modal for manual publishes — both surfaces coexist.

## Dev Notes

### Architecture Constraints

- Outbound webhook is server-only; never invoke from a client component. Use server actions or the cron endpoint.
- Use a DIFFERENT JWT key pair from K2's read-direction. ADR-018 §2 says "audience claim distinguishes them" — using different signing keys is a stronger form of the same principle.
- Idempotency on the receiver side (LINE bot K.12 AC #7) is keyed on `correlation_id`. We MUST NOT generate a new `correlation_id` per retry — reuse the row's existing one.
- Mock mode short-circuit: when `isMockMode()`, the action inserts the `crm_event_log` row but skips outbound delivery; `delivered_at` is left null with `last_error = 'mock-mode-skip'`. Tests should cover this.
