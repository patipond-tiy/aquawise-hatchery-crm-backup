# Story G1: Bind a Customer's LINE Account to Their CRM Record

Status: ready-for-dev

## Story

As a Rep,
I want to mint a one-shot bind link for a customer and send it to them via LINE,
so that their LINE user_id gets connected to their farm record and future Flex messages can reach them.

## Acceptance Criteria

1. On the customer card, a "Connect LINE" CTA is visible when `customer.line_id IS NULL`
2. Clicking the CTA calls a server action that mints a `customer_bind_tokens` row (26-char ULID, 7-day expiry) and returns a LIFF link `liff.line.me/{liffId}/bind?token=...`
3. The rep copies the link and sends it to the farmer via their current channel
4. Farmer opens the link in LINE — the LIFF bind page (in the bot worker repo) authenticates with LINE Login, POSTs to CRM `/api/line/bind` with `{token, lineUserId, lineProfile}`, server consumes the token, sets `customers.line_id`, upserts `line_users`, creates a `chat_threads` row, and deletes the token (one-shot)
5. Token reuse after consumption returns a 409 error; expired tokens (>7 days) return a 410
6. Customer card updates to show a "LINE connected" state and last-seen timestamp

## Dev Notes

### Architecture Constraints

- All API calls via `lib/api/index.ts` facade — never Supabase directly from pages
- `app/api/line/bind/route.ts` is a Next.js API Route (not a Server Action) because the consume path is called by the bot worker service over HTTP, not by a browser form
- The mint path (`POST`) is wrapped as a server action callable from the customer card component
- `'use client'` opt-in only; server components by default
- No `any` in app code; `as` casts allowed only in adapters
- CRM never imports `@line/bot-sdk` or calls LINE Messaging API directly

### Cross-Service Boundary

**CRM-side (this repo):**
| File | Role |
|------|------|
| `app/api/line/bind/route.ts` | POST: mint token; POST (with body `{token, lineUserId, lineProfile}`): consume token (called by bot worker) — **both paths are POST, not GET** (consuming is a mutating operation) |
| `app/[locale]/(dashboard)/customers/actions.ts` | Server action `mintBindLink(customerId)` |
| `components/` (customer card) | "Connect LINE" CTA, conditional on `line_id == null` |
| `lib/mock/api.ts` | Mock `mintBindLink()` returning fake LIFF URL |
| `tests/line/bind.test.ts` | Token expiry, reuse, consume coverage |

**Bot worker repo (separate — `aquawise-line-bot`, Cloud Run `asia-southeast1`):**
- `/liff/bind` LIFF page: LINE Login → POST to CRM `/api/line/bind` → success/error UI
- `LIFF_ID`, `LINE_CHANNEL_ACCESS_TOKEN`, `LINE_CHANNEL_SECRET` env vars live here — not in CRM

### Schema

`customer_bind_tokens` already exists in `supabase/migrations/006_line_integration.sql`. No new migration needed. Confirm the column shape before coding:
- `id` — 26-char ULID (not UUID)
- `customer_id` — FK to `customers`
- `hatchery_id` — FK to `hatcheries` (for RLS scoping)
- `expires_at` — `timestamptz`, set to `now() + interval '7 days'`
- `consumed_at` — `timestamptz`, nullable; set on consumption

### RBAC

- `counter_staff` can mint bind links (`customer:write` action)
- Auditor cannot mint (`customer:write` not in auditor permission set)
- Consume path uses service-role (called by bot worker, not a browser session); RLS bypassed intentionally
- Use `can(role, 'customer:write')` from `lib/rbac.ts` before calling the mint action

### RLS Tables

| Table | Operation | Who | Migration |
|-------|-----------|-----|-----------|
| `customer_bind_tokens` | INSERT (mint) | `hatchery_members` with `customer:write` | `006_line_integration.sql` |
| `customer_bind_tokens` | SELECT + UPDATE `consumed_at` (consume) | service-role | `006_line_integration.sql` |
| `customers` | UPDATE `line_id` | service-role (consume path) | `001_init.sql` |
| `line_users` | UPSERT | service-role | `006_line_integration.sql` |
| `chat_threads` | INSERT | service-role | `006_line_integration.sql` |
