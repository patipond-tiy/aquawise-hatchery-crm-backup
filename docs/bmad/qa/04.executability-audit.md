# Story Executability Audit ‚Äî 2026-05-15

> **What this is.** A second-pass audit, orthogonal to `01.qa-user-story-audit.md`. That one checked story‚Üîbusiness-plan *alignment* (is this the right thing to build). This one checks *executability*: will an autonomous BMAD dev agent succeed following the story's exact file paths, table names, column names, RPC names, RLS citations, and RBAC actions against the **actual codebase** (`supabase/migrations/001`‚Äì`012`, `lib/`, `app/`)?
>
> **Method.** 7 parallel auditors, one per epic chunk, each verifying every concrete reference with `grep`/`ls`/`Read` against the live tree. Same rigor as the BMAD smoke proof (A1/B3), applied to all 33 stories.
>
> **Why it matters.** The "docs ‚Üí agents ‚Üí human QA" model breaks if agents hit doc bugs at execution time ‚Äî humans get pulled back in to debug spec failures instead of QAing output. This audit is the last doc-readiness gate before flipping the agent-dev switch.

---

## Headline findings

1. **The FIX-AUTO pass introduced a BLOCKER.** `01.qa-user-story-audit.md`'s 2026-05-13 batch renamed `batch_buyers` ‚Üí `batch_distributions` in C3 "to fix table-name drift." The actual table is `batch_buyers` (`001_init.sql:97`). `batch_distributions` exists nowhere. The "fix" pointed the story at a non-existent table. **Lesson: a mechanical rename pass with no codebase verification can make stories *less* executable. Every rename must be grep-verified against migrations.**

2. **Systemic migration-number collision.** Six stories each independently claim migration `013`: D2 (`013_quotes.sql`), B3 (`013_customer_history.sql`), E3 (`013_alert_resolutions.sql`), H2 (`013_data_exports.sql`), H4 (`013_quiet_hours.sql`), K1 (`013_batch_code_and_claims.sql`). If more than one lands, numbers collide and `supabase db push` ordering breaks. **Must assign a global migration-number plan before any of these execute.**

3. **`audit_log` schema mismatch breaks K3 and any audit-writing story.** Stories reference `audit_log` columns `event`, `actor`, `subject_id`. Actual schema (`001_init.sql:177-185`): `id, nursery_id, user_id, action, payload, created_at`. Any story that writes audit_log with the assumed columns fails at runtime. (This also affects the `// TODO(audit-log)` guidance in `code-design.md` ¬ß5/¬ß8 ‚Äî the helper spec there must match the real columns.)

4. **`alert_severity` enum has no `'critical'`.** Enum is `('high','medium','low')` (`001_init.sql:121`). E4 AC#5 + Task 6 branch on `severity = 'critical'` ‚Äî the Epic-K critical-warning fan-out silently never fires.

S9's ESLint `getSession` allowlist targets the phantom path ‚Äî it must point at the real cookie-refresh call site or the ban protects nothing.

### SYS-5 ‚Äî No ESLint config exists at all (BLOCKER, S9)
S9 says "confirm `eslint.config.mjs` exists; if not migrate from `.eslintrc.json`." Neither exists. `next lint` runs with Next's built-in defaults only. S9 must be rewritten: "create `eslint.config.mjs` from scratch" ‚Äî materially more work than the story implies.

### Epic C ‚Äî highest risk (3 BLOCKERs, one FIX-AUTO-introduced)

**C2 ‚Äî impossible join.** Story computes mean-D30-per-batch via `customer_cycles.batch_id`. `customer_cycles` has no `batch_id` (`001_init.sql:66-75`: `customer_id PK, cycle_day, expected_harvest, d30, d60, restock_in, updated_at`). There is no FK path cycle‚Üíbatch. **Fix needs a schema decision, not a doc edit:** either (a) add `batch_id` to `customer_cycles` in a migration, or (b) re-spec C2 to use `batch_buyers.d30` (the denormalised per-(batch,customer) D30 that *does* exist). Recommend (b) ‚Äî least scope.

**C3 ‚Äî FIX-AUTO-introduced BLOCKER.** Story says buyers come from `batch_distributions` with columns `quantity`, `sold_at`. Reality: table is `batch_buyers` (`001_init.sql:97`), columns `pl_purchased` (not `quantity`), no `sold_at`. Also references `customer_cycles.batch_id`/`started_at` (don't exist). **Fix: revert the FIX-AUTO rename ‚Äî restore `batch_buyers`, use `pl_purchased`, drop `sold_at`, re-spec the D30 source to `batch_buyers.d30`.**

**C1 ‚Äî WARN.** `pcr:write` RBAC action doesn't exist; C1 self-documents adding it. Sequence C1's rbac.ts edit before its gates compile.

## Process recommendation

Add to `code-design.md` ¬ß16 (review checklist) and the BMAD story-authoring flow: **every concrete reference in a story (migration file, table, column, RPC, RBAC action, source path) must be grep-verified against the live codebase at authoring time, and re-verified by the executability audit before the story is marked ready-for-dev.** Alignment audits (`01`) and executability audits (this doc) are different gates ‚Äî both are required. The FIX-AUTO incident proves an alignment-only pass can ship execution-breaking bugs.
