# Story K5: List Active Batch Codes (LINE-bot nightly sync)

Status: ready-for-dev

> Cross-product gap closed. Conforms to `docs/aquawise-updated-docs/K-INTEGRATION-CONTRACT.md` §5/§8. Required by line-bot K1's nightly batch-code sync (line-bot `decisions-2026-05-15-fix-review.md` ~line 101: *"hatchery-CRM K2 must expose a 'list all active batch codes' endpoint… confirm before line-bot K1 lands"*). K2 is single-code only; this story adds the list endpoint.

## Story

As the LINE bot's nightly sync job,
I want one authenticated call that returns every currently-claimable batch code,
so that I can pre-seed the bot's local index without the bot caching batch metadata across claims (line-bot ADR-018 C-18).

## Acceptance Criteria

1. **Route:** `GET /api/v1/batches?active=true` at `app/api/v1/batches/route.ts`.
2. **Auth:** read-side JWT per K-INTEGRATION-CONTRACT §3 (`iss=line-bot`, `aud=nursery-crm`, ES256, ≤15-min, verified with `CRM_JWT_PUBLIC_KEY`). Missing/invalid/expired → 401, identical semantics to K2 (reuse the same verifier helper K2 creates under `lib/jwt/`).
3. **Query contract:** only `?active=true` is supported in v1.1; any other query (or omitting it) → 400 `{ error: 'unsupported_query' }`. (Forward-compatible: future filters appended, not breaking.)
4. **200 body:** `{ "batches": [ { "batch_code": "B-XXXXXX", "valid_until": "<iso>" }, … ] }` — exactly the codes where `published_at IS NOT NULL AND valid_until > now()`, scoped across all nurseries (this is a platform-internal sync, not tenant-scoped — the bot needs every active code; rows carry no nursery PII, only `batch_code` + `valid_until`). Ordered by `valid_until ASC`. Empty set → `{ "batches": [] }` with 200 (not 404).
5. **No metadata leak:** the response carries ONLY `batch_code` + `valid_until`. No nursery name, contact, PCR, or claim state (those require the single-code K2 call).
6. **Observability:** log `{ correlation_id (from X-Correlation-Id or fresh), count, latency_ms }` per K-INTEGRATION-CONTRACT §5.
7. **Rate limit:** reuse K2's `lib/jwt/rate-limit.ts` helper — 10 req/s per `iss` (the nightly job calls this once; the limit is abuse protection).

## Dev Notes

### Architecture Constraints
- API route, not a server action — no automatic Origin/Host CSRF; the JWT IS the trust boundary (K-INTEGRATION-CONTRACT §3).
- `'server-only'` not required (route handler runs server-side by definition), but use `lib/supabase/server.ts` `createClient`, not the browser client.
- No new RBAC action — auth is the platform JWT, not a `can()` role (the caller is the LINE bot service, not a nursery member).

### Dependencies
- **Hard predecessor: K2.** This story reuses `lib/jwt/` (verifier + rate-limit) that K2 creates. K5 MUST land after K2. Do not duplicate JWT logic.
- No migration. No `batch_code` minting (K1 owns that).

### RLS / RBAC / i18n
- Deliberately cross-nursery (platform sync). No tenant RLS scoping on this endpoint — justified because the payload is `batch_code` + `valid_until` only (no PII, no commercial data). Documented here so a security review doesn't flag it as a tenant-isolation miss; it is an intentional, minimal-surface platform endpoint.
- No i18n (machine-to-machine JSON).
