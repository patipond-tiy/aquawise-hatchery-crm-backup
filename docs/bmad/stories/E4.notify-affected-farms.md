# Story E4: Notify Affected Farms

Status: draft

## Story

As an Owner,
I want to push a "we're investigating + here's what we found" follow-up to the farms named in an alert,
so that they know we're handling it.

## Acceptance Criteria

1. "ส่งข้อความถึงฟาร์ม" opens a template selector with three options: `acknowledge`, `remediation_plan`, `closure`
2. Submit fans out to `line_outbound_events` — one row per affected farm — using template `notifyAlertFarms(alertId, template)`
3. Enqueue is idempotent on `(customer_id, payload->>'alert_id')` — the partial unique index `line_outbound_events_alert_dedupe_idx` already exists in migration 006 (JSONB expression index, NOT a plain column); a second submit for the same alert does not create duplicate events
4. Farms without a bound LINE account (`customers.line_id IS NULL`) are skipped silently; a count of skipped farms is returned to the UI
5. **(LINE bot Epic K integration)** When the alert is linked to one or more `batches` via the `alert_batches` join table (added in this story — see Task 6), and the alert's `severity = 'critical'`, the server action ADDITIONALLY publishes a batch warning per linked batch via the `publishBatchWarning()` server action from Story K4. This is a separate fan-out from the customer (`line_outbound_events`) path in AC #2 — the K-side fans out to *farmers* whose ponds claimed the batch; the customer side fans out to the *nursery* who bought the batch. Both can fire on the same alert. The K-side publish is idempotent (Story K1 `crm_event_log.correlation_id UNIQUE`).
6. **(LINE bot Epic K integration)** UI: when the operator selects the `closure` template AND the alert links to a batch with claimed farmers (`batch_claims` rows exist), the modal surfaces a one-line note: `"ฟาร์ม {n} แห่งที่สแกนรหัสล็อต {code} จะได้รับการแจ้งเตือนนี้ด้วย"`. This makes the auto-publish visible up-front; the operator can cancel before submitting.

## Dev Notes

> **BLOCKED — Do not start implementation until both G3'.i and E3.i are complete.**
>
> G3' (bot worker) is the keystone dependency: without the worker consuming `line_outbound_events`, all enqueued rows will sit at `status='pending'` and farmers will receive nothing. E3 must be complete first so the `closure` template path in E4 is consistent with how alerts are being resolved.

### Domain Framing

> ⚠ **Alert source chain is gated on the farm-side data ecosystem (2027+) — E4 fan-out has no real alerts to fan out from until E2 is unblocked.**
>
> E4 is downstream of E2. E2's AC1 (`farm_cycle_metrics` D30 dip detection) is a **structural pre-2027 dependency**, not a sprint blocker that clears with coordination work. The full dependency chain:
>
> - `farm_cycle_metrics` rows require the farmer-side AquaWise app (น้องน้ำ) to be running at scale.
> - The farmer-side app is stakeholder #1 in the AquaWise sequencing; the hatchery is stakeholder #4 (2027+) — per `docs/aquawise-updated-docs/02-aquawise-what-we-build-first.md` §4: "Approaching P'Bunjong in 2026 with a nursery dataset of n=15 cycles would be selling him a promise, not a product."
> - Until the farm-side data ecosystem reaches 2027 scale, E2 will not generate the `alerts` rows that E4 fans out from. E4 can be implemented against mock `alerts` data for UI/worker completeness, but it has no live alert source until E2 is unblocked.
>
> Additionally, the entire D30 cross-chain feedback loop (hatchery seeing downstream farm survival data) is a **v0.5 hypothesis** pending validation with P'Bunjong — marked ⚠ in `docs/aquawise-updated-docs/06-aquawise-hatchery-customer-doc.md` §2 and §3 Job 2. Do not treat E4's fan-out as confirmed customer behaviour until E2's domain framing is validated.
>
> **For implementation planning:** E4 can proceed as an H2 UI + worker story against manually-created `alerts` rows (e.g., from E1/E3 flows). Its dependency on E2's automated alert source is a live-data concern, not a code-completeness concern. Mark clearly in the task status when E4 is complete against mock alerts vs. against real E2-sourced alerts.

### `alert_farms` Schema Verification

> The `alert_farms` table referenced in AC2/Task 2 as "migration 001 PK (alert_id, customer_id)" **has been verified** in `supabase/migrations/001_init.sql` (lines 141–145):
>
> ```sql
> create table public.alert_farms (
>   alert_id     uuid not null references public.alerts(id) on delete cascade,
>   customer_id  uuid not null references public.customers(id) on delete cascade,
>   primary key (alert_id, customer_id)
> );
> ```
>
> No schema gap. The table exists with the exact PK structure cited in Task 2. The Task 2 step 2 instruction to resolve affected farms via `alert_farms` is correct and implementable without a new migration for this table.

### RLS Tables
| Table | Operation | Migration |
|-------|-----------|-----------|
| `alerts` | SELECT (read affected farms + hatchery scope) | `002_rls.sql` |
| `customers` | SELECT (`line_id` check, hatchery-scoped) | `002_rls.sql` |
| `line_outbound_events` | INSERT (idempotent on `(customer_id, (payload->>'alert_id'))`) | `006_line_integration.sql` |
| `alert_farms` | SELECT (resolve affected farm list) | `001_init.sql` |

The partial unique index `line_outbound_events_alert_dedupe_idx` in migration 006 is on `(customer_id, (payload->>'alert_id'))` — a **JSONB expression index**, not plain columns. The `alert_id` value must be inside the `payload` JSONB object for the ON CONFLICT to work. Do not write `ON CONFLICT (customer_id, alert_id)` — it must be `ON CONFLICT ON CONSTRAINT line_outbound_events_alert_dedupe_idx` or match the expression index exactly.
