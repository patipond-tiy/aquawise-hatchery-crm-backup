# Story B3: View Customer Detail and History

Status: ready-for-dev

## Story

As a Rep (P3 — primary user per persona matrix),
I want to open a customer and see their cycle status, D30 trend, contact info, and batch history,
so that I have full context before calling them.

## Acceptance Criteria

1. Header shows: avatar, farm name, owner, zone, current status chips
2. Stats grid shows: total batches, LTV, last D30, restock-in days — all sourced from real DB rows
3. D30 trend sparkline renders the last 6 cycles' actual D30 values from `customer_cycle_history` (not synthetic)
4. Contact section shows real `phone`, `lineId`, `address` from the `customers` row (no hardcoded literals)
5. Batch history table lists rows from `batch_buyers` joined with `batches` — each row shows cycle status, D30, harvest result chip
6. A customer with zero `customer_cycle_history` rows shows an empty sparkline state (not an error, not dropped)
7. A bad `customer_id` in the URL returns a 404 (Next.js `notFound()`)

## Dev Notes

> **Persona note:** The story was changed from "As an Owner" (legacy `03-user-stories.md#B3`) to "As a Rep (P3)" per the `01-personas.md` persona × page matrix, which marks Rep (★ primary) for Customer detail. Owner retains access (✓) but the primary driver of this workflow is the rep making calls.

### Domain Framing — Critical
The "customer" here is a nursery operator (`โรงอนุบาล`), not a farm. The "batch history" is PL lots distributed to that nursery — not the nursery's own grow-out cycles. The D30 trend reflects the nursery's post-larvae survival rate across batches, which is the primary quality signal the hatchery owner tracks before a call. This distinction matters: `customer_cycles` tracks nursery-side cycles (started when PL distributed), not hatchery-side production cycles. Treat the schema design as a hypothesis pending P'Bunjong validation.

### Architecture Constraints
- All API calls via `lib/api/index.ts` facade — never call Supabase directly from a page

- No `any` in app code; define a `CustomerDetail` type in `lib/types.ts` that includes the sparkline series and history rows

- `can(role, 'customer:read')` — data permission passes for all four roles (`owner`, `counter_staff`, `lab_tech`, `auditor`)

- `batches` — already RLS-scoped; join must not bypass RLS (use security invoker, not definer)
