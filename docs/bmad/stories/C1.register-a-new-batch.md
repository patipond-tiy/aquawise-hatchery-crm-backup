# Story C1: Register a New Batch

Status: ready-for-dev

## Story

As a Manager (P2) or Lab Officer (P4),
I want to register a new PL batch with source strain, spawn date, quantity, and PCR results in under 60 seconds,
so that the batch is ready to assign to buyers and documented with verified quality data.

## Acceptance Criteria

1. A 3-step wizard is presented: (1) batch info, (2) PCR upload, (3) confirm and register.
2. Step 1 collects: source strain (dropdown — options sourced dynamically from a `strains` reference table or whichever source the codebase already provides; the list is not a hardcoded set of 4 strings), spawn date (date picker), and PL count (number input). All fields are required.
3. Step 2 accepts a PDF, JPG, or PNG file drop/select. The file is uploaded to the `pcr-reports/{batch_id}/` Storage bucket. Per-disease results (4 diseases) are captured via a form the user can fill in or override; the current hardcoded "✓ ผ่าน" placeholders are replaced with real controlled inputs.
4. Step 3 shows a confirmation summary of all entered data before final submission.
5. On submit, a `batches` row and one `pcr_results` row per disease are inserted atomically. If the `pcr_results` insert fails the `batches` insert is rolled back.
6. A success toast is shown and the batch list is invalidated (TanStack Query).
7. RBAC enforcement: `counter_staff` may insert a batch row but may NOT insert `pcr_results` rows; `lab_tech` may insert `pcr_results` rows; `owner` may do both. The wizard enforces this at the server action level — not just in the UI.
8. If the PCR file upload fails, the wizard surfaces an error on step 2 and does not advance to step 3.

> **Dependency note:** C1 must be completed before C3 (real buyers/PCR data) and C4 (cert generation) can be fully implemented. These downstream stories are explicitly blocked on C1.i.

## Dev Notes

### Architecture Constraints
- All API calls via `lib/api/index.ts` facade; never call Supabase directly from a page or modal
- Server action lives at `app/[locale]/(dashboard)/batches/actions.ts`
- `'use client'` opt-in only; the modal already uses client state, mark it explicitly
- No `any` in app code; use `Database['public']['Tables']['pcr_results']['Insert']` for the insert type
- Call `requireActiveSubscription()` from `lib/billing/guard.ts` at the top of every mutation server action

### Files to Touch

### API Facade Methods
- `addBatch(input: AddBatchInput): Promise<Batch>` — existing mock stub, needs live implementation
- `uploadPcrReport(batchId: string, file: File): Promise<string>` — new; returns public/signed URL

### RBAC
- `can(role, 'batch:write')` — gates batch row insert (counter_staff, lab_tech, owner)
- **PCR rows require finer-grained gating.** Today `batch:write` includes `counter_staff`, but AC #7 says counter_staff must NOT insert `pcr_results`. Two options:
  1. **(Recommended)** Add a `pcr:write` action to `lib/rbac.ts` granted to `['owner', 'lab_tech']` only. Gate the PCR insert with `can(role, 'pcr:write')`.
  2. Alternatively, hardcode the PCR check as `role === 'owner' || role === 'lab_tech'` in the server action — but this bypasses the `can()` pattern.
- Use `lib/rbac.ts` `can()` function in the server action; do not branch on role strings directly

### RLS Tables
- `batches` — migration 001; `hatchery_id` RLS on insert
- `pcr_results` — migration 001; confirm exact column names before writing insert (see critical note below)
- Storage bucket `pcr-reports` — may need a new migration; check 012

> **CRITICAL:** `pcr_results` is the correct table name. `batch_pcr_tests` does NOT exist and was a documentation drift resolved in migration 001 (see MATRIX.md drift flag). Confirm the column shape by reading migration 001 before writing any insert code.
