# Story S3: File Upload — Validate Magic Bytes Server-Side

Status: done

Severity: P1 — pre-launch hardening

## Story

As an Owner,
I want logo uploads validated by their actual file bytes (not just the client-supplied Content-Type),
so that an attacker cannot upload HTML or polyglot files disguised as images and have them served from our public Storage bucket.

## Acceptance Criteria

1. `lib/supabase/storage.ts:uploadNurseryLogo` reads the first 8 bytes of the upload and rejects anything that is not a known JPEG / PNG / WebP / GIF signature
2. The filename extension is sanitized to a whitelist (`jpg`, `jpeg`, `png`, `webp`, `gif`) — path-traversal characters (`/`, `..`, control chars) are stripped
3. File size is validated in app code (max 2 MB) **before** the upload begins
4. The `Content-Type` passed to Supabase Storage matches the detected magic-byte type — never the client-asserted type
5. A test in `tests/storage/upload.test.ts` covers: valid PNG accepts, HTML-as-PNG rejects, path-traversal filename rejects, oversized file rejects
6. The settings page upload UI surfaces a friendly Thai/English error for each rejection class

## Tasks / Subtasks

## Dev Notes

### Architecture Constraints
- This is a `'server-only'` module — `lib/supabase/storage.ts` already imports `'server-only'`
- Validation runs server-side because client-side checks can be bypassed
- Magic-byte detection is fast (12-byte read) — fine to do synchronously before the upload call
