# Story C2: Browse and Review Batches

Status: ready-for-dev

## Story

As an Auditor (P5),
I want to browse every batch with PCR status visible,
so that I can verify the quality record without seeing commercial information.

## Acceptance Criteria

1. `/batches` shows all this hatchery's batches, newest first (RLS-enforced).
2. Each card shows: batch ID, spawn date, source strain, PCR chip (clean / flagged / pending), produced PL count, buyer count, mean D30.
3. Filter chips for PCR status (clean / flagged / pending), source strain, and year are rendered and bound to URL query state via Next.js `searchParams` ŌĆö selecting a chip updates the URL and triggers a data refetch without a full page reload.
4. Clicking a card navigates to the batch detail page (C3).
5. The `auditor` role sees all batch rows (same RLS read access as owner/manager) but commercial fields ŌĆö LTV, `unit_price`, `sold_at` on distributions ŌĆö are hidden. A SECURITY INVOKER view enforces this at the database level; the application must not rely solely on UI-layer hiding.
6. All other roles (owner, counter_staff, lab_tech) see the full card including commercial fields where present.

## Dev Notes

### Architecture Constraints
- All API calls via `lib/api/index.ts` facade; never call Supabase directly from a page component
- Filter state must live in the URL (`searchParams`) not in React `useState` ŌĆö this keeps the page server-renderable and shareable via link
- `'use client'` opt-in only; the filter chip strip is the only client boundary needed here
- No `any` in app code; use `Database['public']['Views']['batches_auditor_v']['Row']` once the migration is applied and types are regenerated
- Call `requireActiveSubscription()` from `lib/billing/guard.ts` at the top of any server action that performs mutations; read-only list calls do not need the guard

### Files to Touch
- `app/[locale]/(dashboard)/batches/page.tsx` ŌĆö add filter chip strip; read `searchParams`; pass filters to API call
- `lib/api/supabase.ts` ŌĆö extend `listBatches()` with filter params + auditor dispatch
- `lib/api/index.ts` ŌĆö expose updated `listBatches` signature
- `lib/mock/api.ts` ŌĆö match filter logic in mock
- `supabase/migrations/013_auditor_batch_view.sql` ŌĆö new SECURITY INVOKER view
- `lib/database.types.ts` ŌĆö regenerate after migration (`supabase gen types typescript --linked`)
- `tests/batches/list.test.ts` ŌĆö new test file

### API Facade Methods
- `listBatches(hatcheryId: string, filters?: BatchListFilters): Promise<Batch[]>` ŌĆö extend existing; `BatchListFilters = { pcr?: PcrStatus; strain?: string; year?: number }`
- No new write-path methods in this story

### RBAC
- `can(role, 'batch:read')` ŌĆö gates list access (all roles except the enum values without batch:read)
- Auditor sees rows but not commercial columns; enforced at DB view layer AND at app dispatch layer
- Use `lib/rbac.ts` `can()` ŌĆö do not branch on role strings in components

### RLS Tables / Views
- `batches` ŌĆö migration 001; existing RLS on `hatchery_id`
- `batches_auditor_v` ŌĆö new migration 013; SECURITY INVOKER, inherits caller RLS; strips commercial columns
- `batch_buyers` ŌĆö used to compute buyer count (read-only join)
- `customer_cycles` ŌĆö used to compute mean D30 (read-only join)

> **CRITICAL:** Do NOT create a `SECURITY DEFINER` view for the auditor path ŌĆö that bypasses RLS and would expose all hatcheries' rows. Use `SECURITY INVOKER` (the default in Postgres 15+) so the view executes with the calling user's privileges and RLS applies normally.
