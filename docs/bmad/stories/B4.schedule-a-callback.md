# Story B4: Schedule a Callback

Status: done
Phase: **H2**

## Story

As a Rep,
I want to mark "call this farm on Friday at 2pm",
so that I don't forget the next touchpoint.

## Acceptance Criteria

1. "นัดโทร" button on the customer detail page opens the `schedule` modal
2. Modal fields: date (required), time (required), note (optional), channel — call or LINE (required)
3. Submission blocked if `scheduled_for` is in the past
4. On submit: insert one `customer_callbacks` row tagged to the current user (`created_by = auth.uid()`)
5. Callbacks are visible in a new "Upcoming" section on the customer detail page, sorted by `scheduled_for ASC`
6. RLS: `counter_staff` can insert callbacks; only the row owner (`created_by`) or an `owner`-role user can mark a callback completed

## Dev Notes

### Domain Framing — Critical
A callback here is a rep-initiated note to call a specific nursery operator at a specific time. It is NOT a customer-service ticket, a CRM activity log, or a two-way chat thread. The primary value is as a personal reminder for the rep — "don't forget to call P'Bunjong on Friday." The "Inbox view" mentioned in the product spec (AC #5 from `03-user-stories.md`) is explicitly deferred; this story ships only the customer-detail Upcoming section. Notification queuing for the day-of is also deferred — that depends on G4 (cron) and H1 (notification settings).

**Clarification — "call" is a reminder, not a feature:** The channel selector (call / LINE) and the "นัดโทร" label imply a phone call, but there is no VoIP, click-to-call, or in-app calling feature anywhere in the product. The rep makes the call manually outside the app. This story is purely a **reminder tool** — it stores a scheduled reminder with a channel preference. No real call feature is planned or in scope.

### Architecture Constraints
- All API calls via `lib/api/index.ts` facade — never call Supabase directly from a page
- Server action files live next to their routes: `app/[locale]/(dashboard)/customers/[id]/actions.ts`
- Server actions must import `'server-only'` is not required by convention here, but they must NOT export anything that bundles into the client — the file is `'use server'` at the top
- `scheduleCallback` must call `requireActiveSubscription()` from `lib/billing/guard.ts` before any DB write
- Past-date validation must happen server-side (guard in the action) even if also done client-side (inline error in the modal)
- No `any` in app code; define a `CustomerCallback` type in `lib/types.ts`

### API Facade Methods
- `scheduleCallback(input: { customerId: string; nurseryId: string; scheduledFor: Date; channel: 'call' | 'line'; note?: string }): Promise<CustomerCallback>` — new
- `listCallbacks(nurseryId: string, customerId: string): Promise<CustomerCallback[]>` — returns upcoming (incomplete) callbacks sorted by `scheduled_for ASC`
- `completeCallback(callbackId: string): Promise<void>` — sets `completed_at = now()`

### RBAC
- `can(role, 'customer:write')` — governs whether "นัดโทร" is shown; passes for `owner`, `counter_staff`
- `can(role, 'customer:read')` — governs the Upcoming section display; passes for all roles
- Row-level completion: `created_by = auth.uid()` OR role is `owner` — enforced in both the server action and the RLS UPDATE policy

### RLS Tables
- `customer_callbacks` — new table; INSERT for `owner` + `counter_staff` of the same `nursery_id`; SELECT for all roles of same `nursery_id`; UPDATE (mark complete) for row owner or `owner` role
