# Story C2: Browse and Review Batches

Status: done

## Story

As a Manager (P2 — primary user per persona matrix),
I want to browse every batch with PCR status visible,
so that I can track batch quality and distribution status at a glance.

## Acceptance Criteria

1. `/batches` shows all this nursery's batches, newest first (RLS-enforced).
2. Each card shows: batch ID, spawn date, source strain, PCR chip (clean / flagged / pending), produced PL count, buyer count, mean D30.
3. Filter chips for PCR status (clean / flagged / pending), source strain, and year are rendered and bound to URL query state via Next.js `searchParams` — selecting a chip updates the URL and triggers a data refetch without a full page reload.
4. Clicking a card navigates to the batch detail page (C3).
5. All nursery members (owner, counter_staff, lab_tech) see the full card including commercial fields. Auditor-scoped browsing is deferred to Phase H3 alongside the `auditor` persona launch (see `01-personas.md` P5 — "planned, Phase H3"). Per `docs/bmad/decisions-2026-05-15-fix-review.md` §C2, the SECURITY INVOKER auditor view (`batches_auditor_v`) and the role-aware dispatch are NOT built in H1; that work is parked for H3 pickup.

## Dev Notes

### Architecture Constraints
- All API calls via `lib/api/index.ts` facade; never call Supabase directly from a page component
- Filter state must live in the URL (`searchParams`) not in React `useState` — this keeps the page server-renderable and shareable via link
- `'use client'` opt-in only; the filter chip strip is the only client boundary needed here
- No `any` in app code
- Call `requireActiveSubscription()` from `lib/billing/guard.ts` at the top of any server action that performs mutations; read-only list calls do not need the guard

### Files to Touch

### API Facade Methods
- `listBatches(nurseryId: string, filters?: BatchListFilters): Promise<Batch[]>` — extend existing; `BatchListFilters = { pcr?: PcrStatus; strain?: string; year?: number }`
- No new write-path methods in this story

### RBAC
- `can(role, 'batch:read')` — gates list access (owner, counter_staff, lab_tech, auditor — all have batch:read)
- Auditor commercial-field hiding is deferred to H3 (see "Deferred (H3)" above)
- Use `lib/rbac.ts` `can()` — do not branch on role strings in components

### RLS Tables
- `batches` — migration 001; existing RLS on `nursery_id`
- `batch_buyers` — migration 001 (`001_init.sql:97-104`); used to compute both buyer count and mean D30 (read-only join on `batch_id`; columns: `batch_id, customer_id, pl_purchased, d30, created_at`). Do NOT join `customer_cycles` for D30 — that table has no `batch_id` column.
