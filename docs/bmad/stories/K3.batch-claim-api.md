# Story K3: Batch Claim API (`POST /api/v1/batches/:code/claim`)

Status: ready-for-dev

## Story

As **the LINE bot**, after a farmer enters a valid Batch Code in the LIFF pond-setup wizard,
I want to register the claim on the CRM side idempotently,
so that the hatchery operator can see which farms claimed each batch and the LINE bot's local state stays consistent with the CRM source of truth (LINE bot Story K.1 AC #2 + #7).

## Acceptance Criteria

1. **Route shape:** `POST /api/v1/batches/:code/claim` at `app/api/v1/batches/[code]/claim/route.ts`.
2. **Auth:** Same JWT contract as Story K2 (`iss=line-bot`, `aud=hatchery-crm`, ES256, ≤15-min TTL). 401 on failure.
3. **Request body:**
   ```jsonc
   {
     "line_user_id": "U1234...",            // required, regex /^U[0-9a-f]{32}$/
     "pond_id": "p_01H...",                  // required, opaque LINE-side id; stored verbatim
     "line_profile": {
       "display_name": "พี่ปลา",
       "picture_url": "https://..."
     },
     "correlation_id": "<uuid>"             // required, propagated for distributed trace
   }
   ```
4. **Validation:** Missing or malformed field → 400 with `{ error: 'invalid_body', field: '<name>' }`. `correlation_id` must be a valid UUID v4.
5. **Code resolution:** Same ladder as Story K2 — unknown code 404, unpublished 404, expired 410.
6. **Happy path:** INSERT into `batch_claims` `(batch_id, line_user_id, pond_id, line_profile, correlation_id, hatchery_id, claimed_at=now())`. Respond 200 with `{ ok: true, batch_code, claimed_at }`.
7. **Idempotency:** PK `(batch_id, line_user_id)`. Re-POST with the same `(batch_code, line_user_id)` and same `correlation_id` → 200 with the existing row's `claimed_at` (no error). Re-POST with the same `(batch_code, line_user_id)` but a DIFFERENT `correlation_id` → still 200 (treated as retry); the original `correlation_id` and `claimed_at` are preserved (do not overwrite).
8. **Conflict:** If the `(batch_id)` already has a `batch_claims` row for a DIFFERENT `line_user_id`, respond 409 with `{ error: 'claimed_by_other' }`. Do NOT leak the other user's `line_user_id`, `display_name`, or `picture_url`. (Phase 1.1 enforces one-claim-per-batch; this is FR-11b on the bot side.)
9. **Rate limit:** Same 10 req/s per `iss` as K2 (the same `rate-limit.ts` helper is reused).
10. **Audit trail:** On every successful claim (including idempotent re-claims), append a row to the existing `audit_log` table (introduced in `001_init.sql`). The real columns are `(id, hatchery_id, user_id, action, payload, created_at)` — there is **no** `event`, `actor`, or `subject_id` column. Insert with: `action='batch_claim'`, `user_id=NULL` (the LINE bot is not an `auth.users` row; actor identity is embedded in `payload` instead), `hatchery_id` from the batch row, `payload=jsonb_build_object('line_user_id', $line_user_id, 'correlation_id', $correlation_id, 'batch_id', $batch_id, 'actor', 'line-bot:'||$iss)`. Idempotent re-claims insert `action='batch_claim_repeat'` with the same payload shape.
11. **Side effects:** After a successful first claim (NOT a repeat), set `batches.first_claimed_at = now()` if NULL. This column is added to `batches` by migration `013_batch_code_and_claims.sql` (Story K1 AC #1) — confirm K1 has landed before implementing K3. This is a soft signal used by the CRM dashboard ("batch is in the wild") and by Story C3 mod's Farm Claims panel.

## Dev Notes

### Architecture Constraints

- This route lives at `app/api/v1/batches/[code]/claim/route.ts` — note the nested dynamic segment (Next.js App Router supports nested `[param]`/literal segments without issue; confirm against the framework version actually pinned in `package.json`).
- Service-role Supabase client. RLS is bypassed; the RPC enforces hatchery scope by virtue of looking up `batches.hatchery_id` and denormalising it onto `batch_claims.hatchery_id`.
- No `BillingGate` here — same reasoning as K2.
- Atomicity matters: a partial INSERT (claim row written but audit_log missed) makes operator audits unreliable. Wrap in RPC for transactional guarantees.
- The endpoint MUST be safe to call multiple times. The LINE bot's retry cron (`/cron/retry-batch-claims`, K.1 Task 3) WILL re-call this endpoint up to 288 times over 24h after a 5xx blip. Idempotency is non-negotiable.
