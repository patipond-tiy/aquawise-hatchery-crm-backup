# Story E3: Close an Alert

Status: ready-for-dev

## Story

As an Owner,
I want to close an alert with a resolution note,
so that the case is documented.

## Acceptance Criteria

1. "ปิดเคส" opens `closeAlert` modal with: (a) a resolution note field (textarea, wired), (b) multi-select follow-up actions taken
2. Submit calls `closeAlert(id, note, actions)` which sets `closed = true` and appends an `alert_resolutions` row
3. Optionally fires a follow-up Flex message to affected farms (checkbox in modal)
4. Alert disappears from the active list after closing

## Dev Notes

### Architecture Constraints
- All API calls via `lib/api/index.ts` facade — never Supabase directly from pages
- Server actions live at `app/[locale]/(dashboard)/alerts/actions.ts` — create this file if it doesn't exist
- `'use client'` is already set on the modal; the server action is called via `useTransition`
- No `any` in app code; `as` casts allowed only in adapters

### Known Issues (from Verifier finding V3)
Two specific bugs in the current `close-alert-modal.tsx` must be fixed as part of this story:

1. **Single-select dropdown (lines 11–16, 52–61):** The modal uses `<select>` over four fixed `REASONS` strings. The AC requires multi-select follow-up actions. Replace with a checkbox list.
2. **Uncontrolled textarea (lines 65–70):** The note `<textarea>` has no `useState` or `ref` — its value is invisible to the submit handler. Wire it with controlled state.

Both are blocking for any live submission to work correctly.

### RBAC
- Action: `alert:close`
- Permitted roles (per `lib/rbac.ts`): `owner`, `counter_staff` — confirm in `lib/rbac.ts` before shipping; `lab_tech` and `auditor` should not be able to close alerts
- Guard in the server action: `if (!can(role, 'alert:close')) throw new Error('Forbidden')`
- Guard in the modal submit button: `disabled={!can(role, 'alert:close')}`
