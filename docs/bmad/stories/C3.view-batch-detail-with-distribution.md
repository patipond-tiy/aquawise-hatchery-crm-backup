# Story C3: View Batch Detail with Distribution

Status: ready-for-dev

## Story

As a Manager (P2),
I want to see who bought from a specific batch and their D30 outcomes,
so that if a complaint comes in I have full context on lineage and performance.

## Acceptance Criteria

1. The batch detail page header shows: batch ID, PCR chip (clean / flagged / pending), spawn date, source strain.
2. A stats row shows: produced PL count, sold PL count, buyer farm count, mean D30. All four values are computed from real data — not hardcoded.
3. The PCR section shows one row per disease with: disease name, pass/fail result, lab name (from `pcr_results.lab` — NOT hardcoded "DOFR"), and test date. If no `pcr_results` rows exist for the batch, render an empty state.
4. The buyers table shows real rows from `batch_buyers` joined with `customers` — not synthetic data. Columns: farm name, PL purchased (from `batch_buyers.pl_purchased`), D30 (from `batch_buyers.d30`). There is no `sold_at` column on `batch_buyers` — do not render a sold-date column. If no `batch_buyers` rows exist, render an empty state.
5. A D30 distribution histogram (10 bins, equal width from 0% to 100%) is rendered using the `d30` values from `batch_buyers` rows for this batch (`001_init.sql:101`: `batch_buyers.d30 int`). Do NOT source D30 from `customer_cycles` — that table has no `batch_id` column and cannot be joined to a batch. If fewer than 3 data points are available the histogram renders a graceful empty state ("ข้อมูลไม่เพียงพอสำหรับการแสดงกราฟ") rather than a misleading chart. This is the most uncertain feature per source doc §3 Job 2 — the histogram must degrade gracefully; it must not block the page from loading.
6. The page is only accessible after C1 is complete (real `batches` and `pcr_results` rows exist). An invalid or non-existent `id` returns a 404.
7. **(LINE bot Epic K integration)** A new "ฟาร์มที่สแกนรหัสล็อต" panel is rendered below the buyers table. It reads from `batch_claims` (Story K1) and shows: LINE display name (from `line_profile.display_name`), `claimed_at` formatted as Thai date, optional avatar (`line_profile.picture_url`), and a `pond_id` hash truncated to 8 chars. Empty state: `"ยังไม่มีฟาร์มสแกนรหัสล็อตนี้"`. Hide the panel entirely if the batch has `published_at IS NULL` (the batch is not yet claimable, so empty-state would be misleading).
8. **(LINE bot Epic K integration)** A "เผยแพร่ตอนนี้" / "หยุดเผยแพร่" toggle is rendered in the page header next to the PCR chip when `published_at IS NULL` / `published_at IS NOT NULL` respectively. Toggling ON snapshots hatchery contact (same logic as C1 AC #9) and sets `batches.published_at = now()`. Toggling OFF sets `published_at = NULL` — any in-flight claims already recorded are preserved (they remain visible in the panel), but future `GET /api/v1/batches/:code` calls return 404.
9. **(LINE bot Epic K integration)** A "แจ้งเตือนล็อตนี้" button is rendered in the page header (defined fully in Story K4). It opens the publish-warning modal; on success the page refetches and shows the most recent warning under a new "การแจ้งเตือนล็อตนี้" sub-section that lists `crm_event_log` rows for this `batch_id` (severity badge + title_th + posted_at + delivered_at status).

> **Dependency note:** C3 is explicitly blocked on C1.i. AC #7–#9 additionally depend on Stories K1 (schema) and K4 (publish-warning modal). The `pcr_results` and `batch_buyers` tables must have real rows before this page can be fully exercised. Implement and ship C1 first.

## Dev Notes

### Domain Framing

> ⚠ **Cross-nursery D30 histogram is a v0.5 hypothesis — not a confirmed customer requirement.** `docs/aquawise-updated-docs/06-aquawise-hatchery-customer-doc.md` §3 Job 2 explicitly states: *"This is the most powerful and the most uncertain feature in this document — uncertain because we have not validated whether P'Bunjong wants or trusts this kind of feedback, and because the data analysis quality required to make it meaningful is high."* The D30 distribution histogram (AC5) must be implemented with graceful degradation (as specified), but no investment in D30 data-pipeline sophistication is warranted until P'Bunjong validation occurs in 2027. Treat AC5 as a placeholder that must not degrade the page; treat any cross-nursery lineage analysis as deferred past H1.

### Architecture Constraints
- All API calls via `lib/api/index.ts` facade; never call Supabase directly from a page component
- `'use client'` opt-in only; the histogram and any interactive chart controls need client marking, but the page layout and data fetch are server components
- No `any` in app code; extend `lib/types.ts` with a `BatchDetail` type that includes `pcrResults: PcrResult[]` and `buyers: BatchBuyer[]` (sourced from `batch_buyers`; columns: `batch_id, customer_id, pl_purchased, d30, created_at` + joined `customer.name`). No `BatchDistribution` type — use `BatchBuyer`.
- `pcr_results` is the correct table name — `batch_pcr_tests` does NOT exist (drift resolved in MATRIX.md 2026-05-02). `pcr_results` columns: `id, batch_id, disease, status, lab, tested_on, file_url, created_at` — no `updated_at`.
- The buyer table source is `batch_buyers` (NOT `batch_distributions` — that table does not exist anywhere in the schema). The FIX-AUTO pass on 2026-05-13 incorrectly renamed this; this story has been corrected to match `001_init.sql:97`.
- This page is blocked on C1.i: the page can be implemented and rendered in mock mode, but live data requires real `batches` + `pcr_results` rows from C1
