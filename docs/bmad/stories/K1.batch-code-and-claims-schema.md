# Story K1: Batch-Code, Claims, and CRM-Event-Log Schema

Status: ready-for-dev

## Story

As **product**,
We want CRM schema that mints an unguessable per-batch claim code, records LINE-side farmer claims, and tracks outbound events to the LINE bot,
so that LINE bot stories K.1, K.7, K.12, and K.13 have a stable CRM-side foundation.

## Acceptance Criteria

1. **Migration `013_batch_code_and_claims.sql`** adds the following columns to `batches`:
   - `batch_code text` ‚Äî 6-char alphanumeric `B-XXXXXX`, unique across the table, unguessable (excludes confusables `0`/`O`, `1`/`I`/`l`); generated server-side by a Postgres function `mint_batch_code()`; NOT NULL with a default of `mint_batch_code()`.
   - `valid_until timestamptz` ‚Äî defaults to `created_at + interval '30 days'`. Hatchery operators can shorten via UI but never extend past `created_at + 90 days` (CHECK constraint).
   - `published_at timestamptz` ‚Äî nullable. Only batches with `published_at IS NOT NULL AND valid_until > now()` are claimable from the LINE bot.
   - `hatchery_contact_snapshot jsonb` ‚Äî frozen snapshot of `{ line_oa_id, contact_phone_public, hatchery_name_th }` taken at first publish ‚Äî so a claim payload returns hatchery contact even if the operator edits the profile later. Nullable until publish.
2. **`batch_claims` table** records each LINE-side farmer claim:
   - PK `(batch_id, line_user_id)`
   - `claimed_at timestamptz NOT NULL DEFAULT now()`
   - `line_user_id text NOT NULL`
   - `line_profile jsonb` ‚Äî `{ display_name, picture_url }` snapshot from LINE Login; no other PII
   - `pond_id text` ‚Äî opaque id passed by LINE bot; CRM does not query it, only round-trips it for audit
   - `correlation_id uuid NOT NULL` ‚Äî propagated from LINE bot for distributed-trace correlation
   - `hatchery_id uuid NOT NULL` ‚Äî denormalised for RLS (matches `batches.hatchery_id`)
3. **`crm_event_log` table** records outbound webhook events to the LINE bot (used by Story K4):
   - `id uuid PK default gen_random_uuid()`
   - `correlation_id uuid NOT NULL UNIQUE` ‚Äî idempotency key shared with LINE bot
   - `event_type text NOT NULL` ‚Äî `'batch_warning'` for Phase 1.1; future types append here
   - `batch_id text NOT NULL REFERENCES batches(id)`
   - `severity text NOT NULL CHECK (severity IN ('info','warning','critical'))`
   - `payload jsonb NOT NULL` ‚Äî full webhook body
   - `posted_at timestamptz NOT NULL DEFAULT now()`
   - `delivered_at timestamptz` ‚Äî set on 2xx response from LINE bot
   - `attempts integer NOT NULL DEFAULT 0`
   - `last_error text`
   - `hatchery_id uuid NOT NULL` ‚Äî for RLS
4. **RLS policies** scope `batch_claims` and `crm_event_log` to `hatchery_members.hatchery_id` for SELECT; INSERT is service-role only (claim endpoint K3 and webhook publisher K4 both use service-role).
5. **`mint_batch_code()` Postgres function** generates a 6-char alphanumeric code from the alphabet `ABCDEFGHJKMNPQRSTUVWXYZ23456789` (32 chars, ~2^30 entropy / ~1 in 1 billion), retries on UNIQUE violation up to 5 times, raises if all 5 collide.
6. **Indexes**: `batch_claims(batch_id)`, `batch_claims(line_user_id)`, `crm_event_log(batch_id, severity)`, `crm_event_log(delivered_at) WHERE delivered_at IS NULL` (partial index for retry sweep).
7. **`lib/database.types.ts`** is regenerated and committed with the migration.
8. **`hatcheries` profile columns added** to support the K.13 re-purchase contact and A3 AC #6:
   - `line_oa_id text` ‚Äî nullable, CHECK `(line_oa_id IS NULL OR line_oa_id ~ '^@[a-zA-Z0-9_-]{1,49}$')`
   - `contact_phone_public text` ‚Äî nullable, CHECK `(contact_phone_public IS NULL OR contact_phone_public ~ '^0[0-9]{9}$')`
   Both columns nullable until the operator fills them in via the Settings page (Story A3 mod). Existing RLS on `hatcheries` covers these (inherited).

## Dev Notes

### Architecture Constraints

- This story is a pure migration + types regen. No UI, no server actions, no API. Subsequent stories (K2, K3, K4, C1 mod, C4 mod, A3 mod) consume the schema.
- Migration must be append-only and sequentially numbered (`013` follows `012_storage_logos.sql` ‚Äî confirmed). If a parallel migration lands first, renumber to the next free slot ‚Äî do not edit a merged migration.
- Apply to staging via `supabase db push` first; only after smoke pass apply to prod.
- `text` ID for batches is the project convention (`batches.id` is `text`, not `uuid`). New columns follow that convention where they reference `batches.id`.
