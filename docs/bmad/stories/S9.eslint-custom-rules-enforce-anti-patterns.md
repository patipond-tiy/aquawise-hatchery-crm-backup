# Story S9: ESLint Custom Rules to Enforce ¬ß18 Anti-Patterns

Status: ready-for-dev

Severity: P1 ‚Äî pre-launch (shifts ¬ß18 enforcement left, from review-time to lint-time)

## Story

As a Developer,
I want three ¬ß18 anti-patterns to fail `pnpm lint`,
so that the reviewer never has to mention them and the codebase never quietly accumulates them.

## Acceptance Criteria

1. `pnpm lint` fails when:
   - `getSession` is called outside `lib/supabase/middleware.ts` (the only file where it's legitimate, for cookie refresh)
   - `dangerouslySetInnerHTML` is used anywhere in `app/` or `components/`
   - A role-string equality comparison is used (`role === 'owner'`, `member.role === 'counter_staff'`, etc.) outside `lib/rbac.ts`
2. The current codebase passes the new rules (relies on S6 having landed first)
3. Each rule produces a clear error message pointing the developer at the ¬ß18 anti-pattern it enforces and at the `can()`/server-only alternative
4. Rules are configured via `eslint.config.mjs` (flat config) ‚Äî not `.eslintrc.json`
5. `next lint` continues to work with the same config

## Tasks / Subtasks

- [ ] Task 1 ‚Äî Decision: built-in rules or custom plugin? (AC: #1)
  - [ ] `dangerouslySetInnerHTML` ban: use built-in `react/no-danger` with severity `error`
  - [ ] `getSession` ban: use built-in `no-restricted-syntax` with `CallExpression[callee.property.name='getSession']` AST selector
  - [ ] Role-string ban: same ‚Äî `no-restricted-syntax` with a selector like `BinaryExpression[operator='==='][left.property.name='role'][right.value=/owner|counter_staff|lab_tech|auditor/]`
  - [ ] None requires a custom plugin; flat-config `no-restricted-syntax` covers all three
- [ ] Task 2 ‚Äî Flat config migration check (AC: #4)
  - [ ] Confirm `eslint.config.mjs` (or `eslint.config.js`) exists; if not, migrate from `.eslintrc.json` (ESLint 9+ requires flat config)
  - [ ] `next lint` documentation: https://nextjs.org/docs/app/api-reference/config/eslint
- [ ] Task 3 ‚Äî Add the three rules (AC: #1, #3)
  - [ ] Add `react/no-danger: 'error'` to the React plugin config
  - [ ] Add `no-restricted-syntax` rules with descriptive messages:
    ```js
    {
      selector: "CallExpression[callee.property.name='getSession']",
      message: "Do not call supabase.auth.getSession() for authorization. Use getUser() ‚Äî it validates the JWT server-side. See code-design.md ¬ß18(C) and security.md ¬ß9."
    }
    ```
  - [ ] Role-string rule:
    ```js
    {
      selector: "BinaryExpression[operator=/===|!==/] > MemberExpression.left[property.name='role'] + Literal[value=/^(owner|counter_staff|lab_tech|auditor)$/]",
      message: "Do not branch on role strings. Use can(role, action) from lib/rbac. See code-design.md ¬ß9 and ¬ß18(B)."
    }
    ```
- [ ] Task 4 ‚Äî Override for legitimate files (AC: #1)
  - [ ] Add a file-pattern override allowing `getSession` inside `lib/supabase/middleware.ts`:
    ```js
    {
      files: ['lib/supabase/middleware.ts'],
      rules: { 'no-restricted-syntax': 'off' /* or a targeted version */ }
    }
    ```
  - [ ] Same approach for `lib/rbac.ts` ‚Äî the role-string comparison is allowed inside the matrix file
- [ ] Task 5 ‚Äî Verify codebase passes (AC: #2)
  - [ ] `pnpm lint` ‚Äî must exit 0 after S6 has been merged
  - [ ] If any hit remains: confirm it's in an allowed file (then add the override) or fix the violation
- [ ] Task 6 ‚Äî Document in handbook
  - [ ] Update `code-design.md` ¬ß16 (Code review checklist) Brand section to add: "[ ] `pnpm lint` passes ‚Äî the three ¬ß18 ESLint bans catch most violations automatically"

## Dev Notes

### Architecture Constraints
- Flat config is the only supported format in ESLint 9; if the project is still on `.eslintrc.json`, Task 2 includes migrating
- AST selectors are syntactic, not semantic ‚Äî there will be false positives (e.g., `someOtherObject.role === 'owner'` would trip the rule). Tighten the selector pattern if false-positive count is high after Task 5
