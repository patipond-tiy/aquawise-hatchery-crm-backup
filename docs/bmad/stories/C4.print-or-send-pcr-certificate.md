# Story C4: Print or Send PCR Certificate

Status: done

## Story

As a Manager (P2),
I want to print a PCR certificate as a PDF or send it as a LINE Flex Message to the buyer,
so that the farm has documented proof of the lot's quality.

## Acceptance Criteria

1. "พิมพ์ใบรับรอง" triggers server-side PDF generation and initiates a browser download. The PDF is rendered using `@react-pdf/renderer` (not Puppeteer) for Next.js compatibility.
2. The PDF includes: nursery logo, nursery name, batch ID, spawn date, source strain, per-disease PCR results (disease name, pass/fail, lab name, test date), a "Verified by AquaWise" stamp, and the certificate generation date.
3. Brand-tier typography: Plus Jakarta Sans for Latin text, Noto Sans Thai for Thai text, no emoji anywhere in the PDF. No marketing copy.
4. On successful PDF generation, the cert URL is stored in a new `batch_certs` table for re-download (idempotent: if a `batch_certs` row already exists for this `batch_id` and no `pcr_results` row for this batch has `created_at` greater than the existing `batch_certs.generated_at`, return the existing record rather than re-generating. Re-generation is triggered only when at least one `pcr_results` row has `created_at > batch_certs.generated_at`). Note: `pcr_results` has no `updated_at` column (`001_init.sql:106-115`); use `created_at` throughout.
5. "ส่งใบรับรอง LINE" opens the existing `cert-modal.tsx`, listing the batch's buyers (from `batch_buyers`). The manager selects one or more recipients and submits.
6. On submit, one row per selected recipient is inserted into `line_outbound_events` with template `pcr_certificate` — idempotent on `(batch_id, customer_id)` partial unique index.
7. The LINE send path is **blocked by G3'.i** (bot worker not yet built). The modal submission can be wired and the `line_outbound_events` rows can be inserted, but actual LINE delivery will not occur until G3' lands. This must be clearly communicated in the modal UI ("จะส่งเมื่อระบบ LINE พร้อม" / "Will be sent when LINE system is ready").
8. RBAC: only `owner` and `lab_tech` may generate a certificate (cert generation touches `pcr_results`). `counter_staff` may send an existing cert via LINE but may not generate a new one. `auditor` may not perform either action.
9. **(LINE bot Epic K integration)** When the LINE bot calls `GET /api/v1/batches/:code` (Story K2), the response field `pcr_certificate_url` is a **time-limited signed URL** (1-hour expiry, regenerated per request) — NOT the raw Storage public URL. This lets a farmer fetch the cert from inside the LINE app without holding a Supabase session, while still preventing long-term hot-linking. The signed URL is minted by `lib/supabase/storage.ts` `getSignedPcrCertUrl(batchId, ttlSeconds=3600)` introduced in Story K2.
10. **(LINE bot Epic K integration)** The `pcr_results` payload returned to the LINE bot uses normalised disease keys lowercased (`{ wssv, ehp, ihhnv, tsv }`) and values strictly `'negative' | 'positive' | 'pending'`. This is a stable contract: do NOT add new keys without coordinating with the LINE bot team because the K.7 Flex builder iterates the known set. If a nursery's lab reports an additional disease, omit it from the payload rather than adding a new key — the operator can still see the full PCR rows on the CRM batch detail page.
11. **(LINE bot Epic K integration)** Sending a cert via the existing "ส่งใบรับรอง LINE" modal (AC #5/#6) remains the nursery-customer-facing flow and uses `line_outbound_events` + the eventual LINE bot worker. It is unchanged by Epic K. The K.7 path is automatic: when a farmer claims a batch, the LINE bot reads `pcr_certificate_url` from the K2 response and pushes the Flex directly — no CRM-side enqueue.

> **Dependency note:** C4 is blocked on C1.i (real `pcr_results` rows must exist) and C3.i (distribution data needed for buyer list). The LINE send path is additionally blocked on G3'.i. AC #9 depends on Story K2 (signed URL helper). Implement C1, C3, and K1/K2 first.

## Dev Notes

### Architecture Constraints
- `lib/pdf/cert.tsx` must be `'server only'` — `@react-pdf/renderer` renders on the server; do not import it in any client bundle
- Server action lives at `app/[locale]/(dashboard)/batches/actions.ts` (may already be partially created for C1 — extend, do not create a duplicate)
- `'use client'` opt-in only; `cert-modal.tsx` is already a client component; the print button handler needs client marking if it uses `useTransition`
- No `any` in app code; type the `@react-pdf/renderer` components using their exported types
- Call `requireActiveSubscription()` from `lib/billing/guard.ts` at the top of every mutation server action
- The LINE send path (G3') is in a separate repo (`aquawise-line-bot`, Cloud Run); this story only inserts into `line_outbound_events` — it does not call LINE directly

### API Facade Methods
- `getBatchCert(batchId: string): Promise<BatchCert | null>` — new; checks for existing cert record
- `upsertBatchCert(cert: BatchCertInsert): Promise<BatchCert>` — new; writes cert record after upload
- `enqueueCertSend(batchId: string, customerIds: string[]): Promise<{ enqueued: number }>` — new; inserts `line_outbound_events` rows

### RBAC
- `can(role, 'pcr:write')` — gates `generatePcrCertPdf()` (cert generation requires PCR write access: `owner`, `lab_tech` only). **Note:** `pcr:write` must be added to `lib/rbac.ts` as part of C1 (see C1 RBAC notes). If C1 has not yet landed, use `can(role, 'batch:write')` as a temporary gate but add a hardcoded `role !== 'counter_staff'` check until `pcr:write` exists.
- `sendCertificate()` requires `can(role, 'customer:write')` (grants `owner`, `counter_staff`) **OR** `can(role, 'pcr:write')` (grants `owner`, `lab_tech`). The union of these two covers `owner`, `counter_staff`, and `lab_tech` — excluding `auditor`. Use `can(role, 'customer:write') || can(role, 'pcr:write')` as the gate. Do NOT use `batch:read` alone — it includes `auditor`.
- `auditor` may not generate OR send certificates
- Use `lib/rbac.ts` `can()` — do not branch on role strings directly

### RLS Tables
- `batch_certs` — new table created in migration `020_batch_certs.sql` (SYS-1); `nursery_id`-scoped via `batches` FK chain; write restricted to owner/lab_tech. Does not exist in migrations 001–012 — must be created by this story's migration.
- `line_outbound_events` — migration 006 (`006_line_integration.sql`); existing idempotency indexes: `line_outbound_events_cycle_dedupe_idx` (cycle templates) and `line_outbound_events_alert_dedupe_idx` (disease alerts). A new `line_outbound_events_cert_dedupe_idx` on `(customer_id, payload->>'batch_id')` WHERE `template='pcr_certificate'` must be added in migration 020 for cert send idempotency — no such index exists in 006.
- `batches` — migration 001; read needed to fetch batch detail for PDF
- `pcr_results` — migration 001; read needed to populate cert content
- Storage bucket `pcr-certificates` — provision with prefix `{batch_id}/`; tenant-scoped RLS matching `pcr-reports` pattern

> **CRITICAL LINE DEPENDENCY:** The LINE send path (`sendCertificate()` → `line_outbound_events` → bot worker → farmer LINE) will NOT deliver messages until G3'.i (bot worker) is built and deployed. Do not represent this as "sending" in any user-facing string. Use "จะส่งเมื่อระบบ LINE พร้อม" (calm declarative) — never "กำลังส่ง" (implying immediate delivery).

> **PDF LIBRARY CHOICE:** Use `@react-pdf/renderer` — not Puppeteer, not `html2pdf`, not `jsPDF`. Puppeteer requires a full Chromium install which is incompatible with Next.js server actions in serverless/edge environments. `@react-pdf/renderer` runs purely in Node.js with no browser dependency.
