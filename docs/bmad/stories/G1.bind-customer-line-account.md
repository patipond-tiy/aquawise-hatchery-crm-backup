# Story G1: Bind a Customer's LINE Account to Their CRM Record

Status: done

## Story

As a Rep,
I want to mint a one-shot bind link for a customer and send it to them via LINE,
so that their LINE user_id gets connected to their farm record and future Flex messages can reach them.

> **Scope clarification (LINE bot Epic K integration).** G1 binds a **nursery customer** (`customers.line_id`) — a B2B `โรงอนุบาล` operator that the nursery sells PL to. It is the channel used by stories D2 (quote send), C4 (cert send), E4 (alert notify), and the cron broadcasts in D3/G4. It is NOT the path used by individual **farmers** (พี่ปลา) who claim a batch via LIFF wizard. Farmer claims are written to `batch_claims` by the LINE bot via Stories K2/K3 — there is no CRM-side bind link, no `customers` row, and no `line_id` on `customers` for farmers. The two identity systems coexist deliberately: a nursery is a commercial counterparty; a farmer is an anonymous downstream consumer whose only relationship with the CRM is "I scanned this batch code". Do not conflate them in code or queries.

## Acceptance Criteria

1. On the customer card, a "Connect LINE" CTA is visible when `customer.line_id IS NULL`
2. Clicking the CTA calls a server action that calls `POST /api/line/bind/mint`, which mints a `customer_bind_tokens` row (26-char ULID, 7-day expiry) and returns a LIFF link `liff.line.me/{liffId}/bind?token=...`
3. The rep copies the link and sends it to the farmer via their current channel
4. Farmer opens the link in LINE — the LIFF bind page (in the bot worker repo) authenticates with LINE Login, POSTs to CRM `POST /api/line/bind/consume` with `{token, lineUserId, lineProfile}`, server consumes the token, sets `customers.line_id`, upserts `line_users`, creates a `chat_threads` row (schema placeholder — see Dev Notes), and deletes the token (one-shot)
5. Token reuse after consumption returns a 409 error; expired tokens (>7 days) return a 410
6. Customer card updates to show a "LINE connected" state and last-seen timestamp

## Dev Notes

### Architecture Constraints

- All API calls via `lib/api/index.ts` facade — never Supabase directly from pages
- `app/api/line/bind/mint/route.ts` and `app/api/line/bind/consume/route.ts` are Next.js API Routes (not Server Actions) because the consume path is called by the bot worker service over HTTP, not by a browser form
- The mint path (`POST`) is wrapped as a server action callable from the customer card component
- `'use client'` opt-in only; server components by default
- No `any` in app code; `as` casts allowed only in adapters
- CRM never imports `@line/bot-sdk` or calls LINE Messaging API directly

### chat_threads Row — Schema Placeholder Only

⚠ The `chat_threads` row inserted by `POST /api/line/bind/consume` is a **schema placeholder** for the H3 two-way chat feature (story G3). It is created on bind so the bind flow does not need to be reworked when H3 lands. **H1 does NOT use this row for any chat UX** — per rule#4, H1 is send-only Flex; the persistent thread inbox is deferred to H3. No CRM page reads or renders `chat_threads` in H1.

### Schema

`customer_bind_tokens` already exists in `supabase/migrations/006_line_integration.sql`. No new migration needed. Confirm the column shape before coding:
- `token` — `text PRIMARY KEY`, 26-char ULID. **This is the PK — not `id`.** All INSERT and lookup queries must use `token`, not `id`.
- `customer_id` — FK to `customers`
- `nursery_id` — FK to `nurseries` (for RLS scoping)
- `created_by` — FK to `auth.users`
- `expires_at` — `timestamptz`, set to `now() + interval '7 days'`
- `consumed_at` — `timestamptz`, nullable; set on consumption
- `consumed_line_user_id` — `text`, nullable; stores the LINE user ID on consumption

⚠ **`line_users` and `chat_threads` do NOT exist in any migration** (`006_line_integration.sql` only creates `nursery_brand`, `customer_bind_tokens`, and `line_outbound_events`). Any migration that creates these tables must be added as an explicit new migration file before the consume handler can write to them. Do not assume they exist.

### RBAC

- `counter_staff` can mint bind links (`customer:write` action)
- Auditor cannot mint (`customer:write` not in auditor permission set)
- Consume path uses service-role (called by bot worker, not a browser session); RLS bypassed intentionally
- Use `can(role, 'customer:write')` from `lib/rbac.ts` before calling the mint action

### Sequencing Constraint

G1 must land **before** D2, C4, E4, F3, and G2. Those stories all require a bound `line_id` on the customer to push Flex messages. Without G1, the queue can enqueue events but the bot worker has no `line_user_id` to deliver to.
