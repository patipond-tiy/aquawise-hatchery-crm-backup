# Story H3: Subscribe / Manage Billing

Status: ready-for-dev

## Story

As an Owner,
I want to subscribe to the Pro plan after my 30-day trial and manage my subscription via Stripe,
so that I can continue using the CRM without interruption.

> ⚠ Pricing tier (THB 5,000/mo) and trial length are nursery-side hypotheses per
> `aquawise-updated-docs/06-aquawise-hatchery-customer-doc.md` §5. Validate with P'Bunjong
> before committing to a Stripe price object for the nursery plan.

## Acceptance Criteria

1. Trial state: Settings → Billing shows days remaining + "Subscribe" CTA → redirects to Stripe Checkout (Pro plan, THB 5,000/mo (⚠ v0.5 hypothesis per `aquawise-updated-docs/06-aquawise-hatchery-customer-doc.md` §5; validate with P'Bunjong before Stripe price-object commitment for the nursery plan), 30-day no-card trial)
2. Active state: shows renewal date + "Manage" → redirects to Stripe Customer Portal
3. Past-due state: shows red banner + "Update Payment" → redirects to Stripe Customer Portal
4. Webhook `app/api/webhooks/stripe/route.ts` handles 6 event types and flips `hatcheries.subscription_status` atomically; events are idempotent via `subscription_events` table
5. Trial-expired behavior = **read-only with banner** (not full lockout): reads (customers, batches, alerts, scorecard) remain accessible; all mutations and LINE pushes return `PaywallError` (HTTP 402) caught by `requireActiveSubscription()` in `lib/billing/guard.ts`
6. Banner copy is calm, declarative Thai/English — no urgency, no exclamation marks (per `docs/product-spec/07-brand-and-voice.md` ลูกหลานที่เรียนมา voice)
7. `billing:manage` RBAC action — only `owner` can initiate Checkout / Portal; other roles see the billing state read-only
8. `lib/stripe/server.ts` is `server-only` — the Stripe SDK must never bundle into the client

## Dev Notes

### Architecture Constraints
- All API calls via `lib/api/index.ts` facade — never Supabase directly from pages
- `lib/stripe/server.ts` **must** start with `import 'server-only'` — the Stripe SDK and secret key must never reach the client bundle
- Server actions at `app/[locale]/(dashboard)/settings/billing/actions.ts` (or `settings/actions.ts`)
- `'use client'` opt-in only; server components by default
- No `any` in app code

### Stripe Server-Only Pattern
```ts
// lib/stripe/server.ts — already implemented, DO NOT modify without reading first
import 'server-only'
// Lazy-initializes Stripe instance using STRIPE_SECRET_KEY
// Throws if STRIPE_SECRET_KEY is missing
```

Environment variables required (server-only — never prefix with `NEXT_PUBLIC_`):
- `STRIPE_SECRET_KEY`
- `STRIPE_WEBHOOK_SECRET`
- `STRIPE_PRO_PRICE_ID`

Environment variable used in client-safe success/cancel URLs:
- `NEXT_PUBLIC_APP_URL` — this one IS safe to use in client context but should still be composed server-side for Checkout
