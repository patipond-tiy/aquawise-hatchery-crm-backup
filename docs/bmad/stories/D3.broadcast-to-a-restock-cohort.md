# Story D3: Broadcast to a Restock Cohort

Status: draft

## Story

As an Owner,
I want to message all "restock-this-week" farms with a single template,
so that I don't have to send 14 messages by hand.

## Acceptance Criteria

1. "ส่งข้อความหาทุกคน" opens a confirmation dialog showing the count of farms affected by the current urgency filter
2. Template selector in the dialog: `restock_reminder` (default), `new_batch_announcement`, `promo`
3. On confirm, the server action `broadcastToFarms(filterId, template, payload)` fans out one `line_outbound_events` row per recipient — idempotent via the existing `(customer_id, template, cycle_id)` partial unique index (migration 006)
4. A confirmation toast appears: "ส่งถึง {N} ฟาร์ม" where N matches the actual number of recipients enqueued
5. Events are processed by the bot worker (G3') when it is deployed

## Dev Notes

### Architecture Constraints
- All API calls via `lib/api/index.ts` facade — never Supabase directly from pages
- Server actions at `app/[locale]/(dashboard)/restock/actions.ts`; call `requireActiveSubscription()` before any write
- `'use client'` opt-in only; the confirmation modal must be a client component (`'use client'`); the restock page may remain a server component
- No `any` in app code
- Adding a new modal kind requires three changes: modal union in `lib/store/modal.ts`, branch in `components/modals/ModalRoot.tsx`, and the new component file

### Blocker
This story is **BLOCKED** by:
- **G3'.i** (Send-only Flex messaging worker): the `line_outbound_events` queue exists (migration 006) and the idempotency index is in place, but the bot worker that processes the queue is not yet deployed. Enqueuing rows is safe to implement now — the farmer will not receive messages until the worker goes live.

The CRM-side implementation (Tasks 1–5) can proceed independently. Live end-to-end delivery verification (Task 6 last bullet) requires G3' to be deployed.

### API Facade Methods
`broadcastToFarms` is a server action called directly from the modal component via `useTransition` — it does not go through `lib/api/index.ts`. No facade change is required. In mock mode the action should return a mock count derived from the current in-memory customer list (add a mock stub in `lib/mock/api.ts` if the action imports it, or gate on `USE_MOCK` inside the action).

### RBAC
| Action | Roles permitted |
|--------|----------------|
| Broadcast to cohort | `owner` only — check `can(role, 'broadcast:write')` (added 2026-05-15 per `docs/bmad/decisions-2026-05-15-fix-review.md` §D3 (b)). Dedicated action, not `settings:write` — broadcasts are an outbound comms operation distinct from settings changes. |
| View broadcast button | Hide for `counter_staff`, `lab_tech`, `auditor` — use `can(role, 'broadcast:write')` guard in the page component |

### RLS Tables
| Table | Operation | Migration |
|-------|-----------|-----------|
| `customers` | SELECT (hatchery-scoped, to resolve cohort) | `001_init.sql` / `002_rls.sql` |
| `line_outbound_events` | INSERT (owner only) | `006_line_integration.sql` |

The partial unique index on `line_outbound_events` in migration 006 is on `(customer_id, template, (payload->>'cycle_id')) WHERE status IN ('pending','sending','sent')` — a **JSONB expression index**, not plain columns. The `cycle_id` value must be inside the `payload` JSONB object for the ON CONFLICT to work. Use `ON CONFLICT ON CONSTRAINT line_outbound_events_cron_dedupe_idx` or match the expression exactly. The server action must catch unique-violation errors per customer and continue rather than aborting the entire batch.
