# Story A2: Invite Team Members

Status: ready-for-dev

## Story

As an Owner,
I want to invite my manager, rep, or lab officer with a specific role,
so that they can access the right parts of the CRM without sharing my login.

## Acceptance Criteria

1. "+ เชิญสมาชิก" button opens the `invite` modal
2. Modal fields: email (required), role (counter_staff / lab_tech / auditor), display name (optional)
3. Submit creates a `team_invites` row with a 64-char hex token, 7-day expiry; sends a Supabase Admin invite email with the accept link
4. Recipient clicks link → magic-link sign-in → joined as `nursery_members` with the chosen role; token marked `accepted_at`
5. Only the `owner` role can submit; `counter_staff` and others are blocked at the server action level (not just UI)
6. An expired token (`expires_at < now()`) and an already-used token (`accepted_at IS NOT NULL`) both redirect to an error page rather than granting access

## Dev Notes

### Architecture Constraints
- All API calls via `lib/api/index.ts` facade — never Supabase directly from pages
- Server actions at `app/[locale]/(dashboard)/<page>/actions.ts`; auth-path server routes live at `app/auth/<path>/route.ts`
- `'use client'` opt-in only; server components are the default
- No `any` in app code; `as` casts allowed only in adapters

### Files to Touch
The implementation (`.i`) is already complete (commit `56a9492`). The following files should be read during verification — no code changes are expected unless a gap is discovered.

| File | Role |
|------|------|
| `components/modals/invite-team-modal.tsx` | Client modal; submit wired with `useTransition`; renders role picker |
| `app/[locale]/(dashboard)/settings/team/actions.ts` | Server action; validates email + role + owner; inserts `team_invites`; calls `admin.inviteUserByEmail` |
| `app/auth/accept-invite/route.ts` | GET route; validates token; upserts `nursery_members`; stamps `accepted_at` |
| `supabase/migrations/008_team_invites.sql` | Creates `team_invites` table, indexes, and owner-only RLS policy |
| `supabase/migrations/011_rls_tighten.sql` | Tightens `nursery_members` INSERT/DELETE and `subscription_events` SELECT to owner-only |
| `tests/team/invite.test.ts` | 7 Vitest cases covering validation, token shape, and access control |

### API Facade Methods
No new `lib/api/index.ts` facade methods are needed for A2. `inviteTeamMember` is a server action called directly from the modal — it does not go through the facade because it needs the authenticated Supabase client on the server side.

### RBAC
- `can(role, 'team:invite')` returns `true` only for `owner` (see `lib/rbac.ts`)
- The server action enforces this by querying `nursery_members WHERE role = 'owner'` rather than calling `can()` directly; the check is equivalent
- The modal deliberately excludes `owner` from the role picker — invited members can be `counter_staff`, `lab_tech`, or `auditor` only
- `counter_staff` cannot invite; attempting to call the action while logged in as `counter_staff` returns `{ ok: false, error: 'ไม่มีสิทธิ์เชิญสมาชิก' }`

### RLS Tables
| Table | Operation | Migration | Who |
|-------|-----------|-----------|-----|
| `team_invites` | ALL (select, insert, update, delete) | `008_team_invites.sql` | `owner` of the nursery |
| `nursery_members` | INSERT | `011_rls_tighten.sql` | `owner` only |
| `nursery_members` | DELETE | `011_rls_tighten.sql` | `owner` only |
| `subscription_events` | SELECT | `011_rls_tighten.sql` | `owner` only |

The `team_invites_all` policy uses a sub-select against `nursery_members WHERE role = 'owner'` in both `USING` and `WITH CHECK` clauses — so even a service-role bypass is not needed for the insert path, only for `admin.inviteUserByEmail`.

### Token Shape
- `generateToken()` returns `randomBytes(32).toString('hex')` — a 64-character hex string
- Token is stored as `text NOT NULL UNIQUE` in `team_invites`
- The accept URL format is `{NEXT_PUBLIC_APP_URL}/auth/accept-invite?token={token}`
- Token is single-use: once `accepted_at` is stamped the route rejects further use

### Email Service
- Production path: `admin.inviteUserByEmail(email, { redirectTo: acceptUrl })` via `createServiceClient()` — requires `SUPABASE_SERVICE_ROLE_KEY`
- Dev fallback: when `SUPABASE_SERVICE_ROLE_KEY` is unset the action logs the accept URL to the console and still returns `{ ok: true }` so the invite row is saved
- If the email send throws (e.g. rate limit), the error is caught silently and the action still returns `{ ok: true }` — the invite row is in the DB and can be resent later
- Resend / SMTP configuration is a separate ops concern; the action will use whatever email provider is wired into the Supabase project
