# Story A1: Sign Up and Create a Hatchery Workspace

Status: ready-for-dev

## Story

As an Owner,
I want to create my hatchery workspace from a magic-link email,
so that I can start using the CRM without a password.

## Acceptance Criteria

1. Email entered on `/login` → magic link sent via Supabase Auth
2. Click link → `/auth/callback` exchanges code → redirected to `/{locale}`
3. First-ever sign-in for the email creates a `hatcheries` row (name from email domain placeholder), a `hatchery_members` row with role `owner`, and a 30-day Stripe trial subscription via the `create_hatchery` RPC
4. Subsequent sign-ins land directly on the dashboard without repeating workspace creation

## Dev Notes

### Architecture Constraints
- All API calls via `lib/api/index.ts` facade — never Supabase directly from pages
- Server actions at `app/[locale]/(dashboard)/<page>/actions.ts`; auth-path server actions live at `app/auth/callback/actions.ts`
- `'use client'` opt-in only; server components by default
- No `any` in app code; `as` casts allowed only in adapters

### Files to Touch
The implementation (`.i`) is already complete (commit `56a9492`). The following files should be read during verification — no code changes are expected unless a gap is discovered.

| File | Role |
|------|------|
| `app/auth/callback/actions.ts` | Server action that calls bootstrap after PKCE exchange |
| `app/auth/callback/page.tsx` | Client page that invokes the callback action |
| `lib/auth/bootstrap.ts` | Idempotent workspace-creation logic; calls `create_hatchery` RPC |
| `lib/utils/mock-mode.ts` | Short-circuit guard used by bootstrap in mock mode |
| `supabase/migrations/001_init.sql` | `hatcheries`, `hatchery_members` table definitions |
| `supabase/migrations/002_rls.sql` | RLS policies for `hatcheries` / `hatchery_members`; `create_hatchery` RPC is defined here (NOT in `001_init.sql`) |
| `supabase/migrations/004_billing.sql` | `create_hatchery` RPC extended with `trial_ends_at = now() + interval '30 days'` billing fields; also contains the authoritative RPC body |
| `tests/auth/bootstrap.test.ts` | Vitest unit tests for bootstrap idempotency |

### API Facade Methods
No new facade methods are required for A1. Bootstrap is invoked server-side directly from the callback action, not via the `lib/api` facade.

### RBAC
- Bootstrap always creates the first user as `owner`; no `can()` check is needed inside the bootstrap itself
- Subsequent page loads use `can(role, action)` from `lib/rbac.ts` — no changes needed for A1

### RLS Tables
| Table | Operation | Migration |
|-------|-----------|-----------|
| `hatcheries` | INSERT (via `create_hatchery` RPC, service-role) | `002_rls.sql` / `004_billing.sql` |
| `hatchery_members` | INSERT (via `create_hatchery` RPC, service-role) | `002_rls.sql` / `004_billing.sql` |
| `notification_settings` | INSERT (via `create_hatchery` RPC) | `002_rls.sql` / `004_billing.sql` |
| `scorecard_settings` | INSERT (via `create_hatchery` RPC) | `002_rls.sql` / `004_billing.sql` |

All RLS policies for these tables are in `002_rls.sql`. The `create_hatchery` RPC is defined in `002_rls.sql` and extended in `004_billing.sql` (adds `trial_ends_at`); it was **not** defined in `001_init.sql`. The function runs with elevated privileges; end-user RLS is not exercised during the insert path.
