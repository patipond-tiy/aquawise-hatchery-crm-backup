# Story C2: Browse and Review Batches

Status: ready-for-dev

## Story

As a Manager (P2 тАФ primary user per persona matrix),
I want to browse every batch with PCR status visible,
so that I can track batch quality and distribution status at a glance.

## Acceptance Criteria

1. `/batches` shows all this hatchery's batches, newest first (RLS-enforced).
2. Each card shows: batch ID, spawn date, source strain, PCR chip (clean / flagged / pending), produced PL count, buyer count, mean D30.
3. Filter chips for PCR status (clean / flagged / pending), source strain, and year are rendered and bound to URL query state via Next.js `searchParams` тАФ selecting a chip updates the URL and triggers a data refetch without a full page reload.
4. Clicking a card navigates to the batch detail page (C3).
5. All hatchery members (owner, counter_staff, lab_tech) see the full card including commercial fields. Auditor-scoped browsing is deferred to Phase H3 alongside the `auditor` persona launch (see `01-personas.md` P5 тАФ "planned, Phase H3"). Per `docs/bmad/decisions-2026-05-15-fix-review.md` ┬зC2, the SECURITY INVOKER auditor view (`batches_auditor_v`) and the role-aware dispatch are NOT built in H1; that work is parked for H3 pickup.

## Dev Notes

### Architecture Constraints
- All API calls via `lib/api/index.ts` facade; never call Supabase directly from a page component
- Filter state must live in the URL (`searchParams`) not in React `useState` тАФ this keeps the page server-renderable and shareable via link
- `'use client'` opt-in only; the filter chip strip is the only client boundary needed here
- No `any` in app code
- Call `requireActiveSubscription()` from `lib/billing/guard.ts` at the top of any server action that performs mutations; read-only list calls do not need the guard

- `lib/api/index.ts` тАФ expose updated `listBatches` signature
