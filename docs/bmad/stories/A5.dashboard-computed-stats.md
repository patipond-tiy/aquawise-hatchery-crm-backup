# Story A5: Dashboard Computed Stats and Role-Aware Content

Status: ready-for-dev

## Story

As an Owner (P1 — primary user per persona matrix),
I want the dashboard hero stats to show real numbers from my data,
so that I can trust the three KPIs I check every morning without doubting whether they are placeholders.

## Acceptance Criteria

1. "ล็อตที่กำลังเลี้ยง" chip shows `{active}/{total}` computed from `batches` where status = `active` vs. total batch count
2. "D30 อัตรารอดเฉลี่ย" chip shows the mean D30 across all active batches (or `—` if no data), formatted as `{value}%`
3. "ต้องสั่งใหม่ใน 14 วัน" chip shows the count of customers with `restockIn ≤ 14` and their estimated total PL volume
4. "Continue Watching" cards show the customer's actual current batch reference (not hardcoded `'ล็อต B-2604-A'`)
5. Dashboard is accessible to Owner (★), Manager (✓), Rep (✓), Auditor (✓). **Lab (—)** should not see the dashboard — redirect to their primary surface (`/batches`)
6. All stat values degrade gracefully: if no batches exist, show `0/0` or `—`, not an error

  - [ ] Query `customers` joined with `customer_cycles` for `restockIn ≤ 14`: count of customers + SUM of estimated PL volume

## Dev Notes

### Implementation Status
- **A5.i is ❌** — not started. All three stat chips are hardcoded literal strings in the dashboard page. Continue-watching chip text is hardcoded by array index.
- **A5.t is ❌** — `tests/dashboard/stats.test.ts` does not exist.
- **A5.v is ❌** — blocked on A5.i.

### Architecture Constraints
- All API calls via `lib/api/index.ts` facade — never call Supabase directly from a page
- Dashboard page (`app/[locale]/(dashboard)/page.tsx`) is a server component — keep it server-side
- No `any` in app code; define `DashboardStats` type in `lib/types.ts`
- Number formatting: use Thai locale formatting for PL volume (e.g., `620,000`) via `Intl.NumberFormat('th-TH')`

### Files to Touch
- `lib/types.ts` — add `DashboardStats` type
- `lib/api/supabase.ts` — add `getDashboardStats()`, extend `listCustomers()` or add `getContinueWatching()`
- `lib/api/index.ts` — export new facade methods
- `lib/mock/api.ts` — add mock implementations
- `app/[locale]/(dashboard)/page.tsx` — replace all hardcoded stats with real data
- `tests/dashboard/stats.test.ts` — new test file (3 cases minimum)

### API Facade Methods
- `getDashboardStats(hatcheryId: string): Promise<DashboardStats>` — new
- `getContinueWatching(hatcheryId: string, limit?: number): Promise<ContinueWatchingItem[]>` — new (or extend `listCustomers` with a `sort` param)

### RBAC
- `can(role, 'customer:read')` and `can(role, 'batch:read')` — both needed for stats queries
- Page access: `owner`, `counter_staff`, `auditor`. **Redirect `lab_tech`** to `/batches`.
- Per persona matrix: Owner (★), Manager (✓), Rep (✓), Auditor (✓), Lab (—)

### RLS Tables
- `batches` — RLS-scoped; used for active/total count and mean D30
- `customers` — RLS-scoped; used for restock count
- `customer_cycles` — RLS-scoped; used for `restockIn` calculation
- `batch_buyers` — RLS-scoped; used for continue-watching batch reference
