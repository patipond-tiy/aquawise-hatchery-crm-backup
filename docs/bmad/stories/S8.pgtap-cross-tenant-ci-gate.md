# Story S8: pgTAP Cross-Tenant Test as CI Required Check

Status: done

Severity: P0 — required for launch (cross-tenant data leaks are the #1 multi-tenant SaaS incident class)

## Story

As an Owner,
I want every pull request to run an automated test that proves a user from tenant A cannot read or write any row belonging to tenant B,
so that a developer cannot accidentally ship an RLS regression — the most catastrophic class of bug in a multi-tenant SaaS.

## Acceptance Criteria

1. `supabase/tests/cross-tenant.sql` exists and contains pgTAP assertions for every tenant-scoped table in `architecture.md` §5
2. The test creates two Supabase users, gives them separate hatcheries, has user A insert one row per table, and asserts user B sees `count = 0` on every table
3. The test runs locally via `supabase test db`
4. CI workflow `.github/workflows/ci.yml` runs `supabase test db` as a required check; the job fails the PR if any assertion fails
5. Adding a new tenant-scoped table is required to update `cross-tenant.sql` in the same PR (lint guidance in `code-design.md` §8 step 8 + checklist update in §16)
6. A deliberately-broken RLS policy (e.g., `using (true)` on one table) causes the CI check to fail — verified by a one-commit branch that demonstrates the failure mode

## Dev Notes

### Architecture Constraints
- pgTAP is the canonical Supabase test tool — see `code-design.md` §11
- Use transaction-wrapped tests (`BEGIN; …; ROLLBACK;`) so multiple test files don't pollute each other
- The test runs against a freshly-seeded local Supabase, not the production project — never run `supabase test db` against production
