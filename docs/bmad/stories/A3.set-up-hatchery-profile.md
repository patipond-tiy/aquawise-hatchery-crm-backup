## Dev Notes

### Clarification ‚Äî "LINE OA" field is branding metadata, not credentials
The "LINE OA" field in the hatchery profile does NOT mean the hatchery enters their own LINE Official Account ID or API credentials. There is one shared `@aquawise` OA owned by AquaWise. The "LINE OA" field stores branding metadata ‚Äî the hatchery's logo and display name in `hatchery_brand` ‚Äî so that messages sent through `@aquawise` appear **co-branded** (hatchery logo + name in Flex Messages) rather than generic. The hatchery never holds LINE API credentials. See `docs/product-spec/05-line-integration.md` for the full model. The UI label "LINE OA" is potentially misleading; consider clarifying in the settings page copy.

### Clarification ‚Äî `line_oa_id` (AC #6) is different from the brand-metadata field above
The Epic K `line_oa_id` field (AC #6) is a **publishable contact handle** ‚Äî the hatchery's own public LINE OA (e.g., `@somthak-hatchery`) that they manage independently of AquaWise. It is shown to farmers verbatim as a "‡∏ï‡∏¥‡∏î‡∏ï‡πà‡∏≠‡∏Æ‡∏≠‡∏ï‡πÄ‡∏ä‡∏≠‡∏£‡∏µ‡πà" link in the K.13 re-purchase prompt. The CRM never sends messages through it; the farmer's LINE client opens a chat draft to that OA when they tap the button. This is a different concern from the AquaWise-owned shared OA used for Flex sends. Settings page copy should distinguish them clearly to avoid operator confusion.

### ‚ö† Flex Co-Brand Surface ‚Äî Hypothesis Dependency

> **‚ö† v0.5 hypothesis.** AC#5 (brand fields propagate to Flex Message templates) depends on the LINE co-brand surface described in `docs/aquawise-updated-docs/06-aquawise-hatchery-customer-doc.md` ¬ß3 Job 3 ("branded Flex quotation + certificate push"). That surface is explicitly marked ‚ö† in the source doc ‚Äî it is an **unvalidated hypothesis** pending the 2027 P'Bunjong discovery conversation. AC#5 of this story is limited to verifying that the `hatchery_brand` row is correctly written after save. Whether the bot worker actually reads and applies these brand fields in live Flex sends is a G-epic deliverable (`G3p`) and inherits the same ‚ö† status. Do not treat AC#5 as confirming end-to-end co-brand behavior.

### Architecture Constraints
- All API calls via `lib/api/index.ts` facade ‚Äî never Supabase directly from pages
- Server actions at `app/[locale]/(dashboard)/<page>/actions.ts`
- Storage helper is `'server-only'` (`lib/supabase/storage.ts` imports `'server-only'`) ‚Äî never call it from a client component
- `'use client'` opt-in only; server components are the default
- No `any` in app code; `as` casts allowed only in adapters

### API Facade Methods
No new `lib/api/index.ts` facade methods are required for A3. Profile reads go through the existing `getHatchery()` and `getHatcheryBrand()` (or equivalent) facade methods. Profile writes use a dedicated server action to support `audit_log` and billing guard without changing the facade surface.

### RBAC
- `can(role, 'settings:write')` returns `true` only for `owner` (see `lib/rbac.ts`)
- The server action additionally gates logo upload to `role === 'owner'` explicitly: `if (logoFile && role === 'owner') { ... }`
- `counter_staff` can call `updateProfile()` for text fields if the action's permission model is relaxed in future, but they cannot change the logo
- `lab_tech` and `auditor` have no write access to settings

### RLS Tables
| Table | Operation | Migration | Who |
|-------|-----------|-----------|-----|
| `hatcheries` | UPDATE | `011_rls_tighten.sql` | `owner` only |
| `hatchery_brand` | UPSERT (INSERT + UPDATE) | `006_line_integration.sql` | hatchery members (all roles can read; brand update governed by app-layer role check) |
| `storage.objects` in `hatchery-logos` | SELECT | `012_storage_logos.sql` | public (no auth required) |
| `storage.objects` in `hatchery-logos` | INSERT, UPDATE, DELETE | `012_storage_logos.sql` | authenticated users in the same hatchery |

Note: `hatchery_brand` RLS intentionally allows all hatchery members to update brand assets day-to-day. The `owner`-only constraint on logo upload is enforced at the app layer (`role === 'owner'` check in the server action), not at RLS level.

### Storage Details
- Bucket name: `hatchery-logos`
- Public: `true` (logos are served without auth; required for Flex messages and the public scorecard)
- File size limit: 2 MB (`file_size_limit: 2097152`)
- Allowed MIME types: `image/jpeg`, `image/png`, `image/webp`, `image/gif`
- Path layout: `{hatchery_id}/logo.{ext}` ‚Äî deterministic so re-uploads overwrite the previous file
- Upload mode: `upsert: true` ‚Äî no duplicate objects accumulate
- Public URL: retrieved via `supabase.storage.from('hatchery-logos').getPublicUrl(path)`

### `ProfileFields` Interface
```ts
export interface ProfileFields {
  name: string;           // hatcheries.name
  name_en: string;        // hatcheries.name_en
  location: string;       // hatcheries.location
  location_en: string;    // hatcheries.location_en
  display_name_th: string; // hatchery_brand.display_name_th
  display_name_en: string; // hatchery_brand.display_name_en
  brand_color: string;    // hatchery_brand.brand_color
  logoFile?: File | null; // transient; not persisted directly
  // Epic K (AC #6)
  line_oa_id?: string | null;          // hatcheries.line_oa_id
  contact_phone_public?: string | null; // hatcheries.contact_phone_public
}
```

The action accepts `Omit<ProfileFields, 'logoFile'>` plus an optional `logoFile?: File | null` as a second parameter so the file can be passed from a `<form>` action or called directly from a button handler.
