# Story H1: Edit Notification Preferences

Status: draft

## Story

As an Owner, Manager, Rep, or Lab user (all roles except Auditor per persona matrix),
I want to toggle which alerts I receive,
so that I am not flooded with unwanted notifications.

## Acceptance Criteria

1. Settings ‚Üí Notifications shows 6 toggles: restock, lowD30, disease, lineReply, weekly, priceMove
2. Each toggle is optimistic ‚Äî the UI updates immediately on click and rolls back on server error
3. Toggle state persists to `notification_settings` (RLS-scoped to the hatchery)
4. Bot worker / cron reads `notification_settings` at delivery time before pushing any LINE event ‚Äî a toggled-off channel produces no outbound event
5. Roles `owner`, `counter_staff`, `lab_tech` can read notification settings; only `owner` can write. **Exclude `auditor`** ‚Äî Auditor is **‚Äî** in persona matrix for Settings ‚Üí Notifications; a read-only compliance role has no need to configure notification preferences.

## Dev Notes

### Architecture Constraints
- All API calls via `lib/api/index.ts` facade ‚Äî never Supabase directly from pages
- `'use client'` opt-in only; server components by default
- No `any` in app code; `as` casts allowed only in adapters
- Delivery-time check lives in the **bot worker repo** (`aquawise-line-bot`) ‚Äî not in this CRM. Coordinate with the bot-worker implementation (G3'.i) before marking H1.i ‚úÖ

### Why H1.i is üü° (not ‚úÖ)
The 6 toggles persist correctly to `notification_settings` (implemented). The missing piece is the **delivery-time read**: the bot worker and the cron route do not currently fetch `notification_settings` before pushing. Until G3'.i (bot worker) and G4.i (cron) are both implemented, no code path enforces the user's preference at delivery time.

### API Facade Methods
No new facade methods expected. The toggle write is via an existing `updateNotificationSettings()` or equivalent action. The delivery-time check in the bot worker calls Supabase directly (service-role client) ‚Äî it is not mediated by the CRM facade.

### RBAC
- `notification_settings` write is `settings:write` ‚Üí owner only (per `lib/rbac.ts`)
- Read is open to all roles (all users need to see their own notification state)
- Use `can(role, 'settings:write')` in any server action that mutates the row

### RLS Tables
| Table | Operation | Migration |
|-------|-----------|-----------|
| `notification_settings` | SELECT (all roles), UPDATE (owner only via RLS) | `002_rls.sql` |

### i18n Keys
No new i18n keys expected. Existing toggle labels in `messages/th.json` and `messages/en.json` under the `settings.notifications` namespace should already cover the 6 toggles.
