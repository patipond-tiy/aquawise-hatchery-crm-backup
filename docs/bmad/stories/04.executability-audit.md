# Story Executability Audit — 2026-05-15

> **What this is.** A second-pass audit, orthogonal to `01.qa-user-story-audit.md`. That one checked story↔business-plan *alignment* (is this the right thing to build). This one checks *executability*: will an autonomous BMAD dev agent succeed following the story's exact file paths, table names, column names, RPC names, RLS citations, and RBAC actions against the **actual codebase** (`supabase/migrations/001`–`012`, `lib/`, `app/`)?
>
> **Method.** 7 parallel auditors, one per epic chunk, each verifying every concrete reference with `grep`/`ls`/`Read` against the live tree. Same rigor as the BMAD smoke proof (A1/B3), applied to all 33 stories.
>
> **Why it matters.** The "docs → agents → human QA" model breaks if agents hit doc bugs at execution time — humans get pulled back in to debug spec failures instead of QAing output. This audit is the last doc-readiness gate before flipping the agent-dev switch.

---

## Headline findings

1. **The FIX-AUTO pass introduced a BLOCKER.** `01.qa-user-story-audit.md`'s 2026-05-13 batch renamed `batch_buyers` → `batch_distributions` in C3 "to fix table-name drift." The actual table is `batch_buyers` (`001_init.sql:97`). `batch_distributions` exists nowhere. The "fix" pointed the story at a non-existent table. **Lesson: a mechanical rename pass with no codebase verification can make stories *less* executable. Every rename must be grep-verified against migrations.**

2. **Systemic migration-number collision.** Six stories each independently claim migration `013`: D2 (`013_quotes.sql`), B3 (`013_customer_history.sql`), E3 (`013_alert_resolutions.sql`), H2 (`013_data_exports.sql`), H4 (`013_quiet_hours.sql`), K1 (`013_batch_code_and_claims.sql`). If more than one lands, numbers collide and `supabase db push` ordering breaks. **Must assign a global migration-number plan before any of these execute.**

3. **`audit_log` schema mismatch breaks K3 and any audit-writing story.** Stories reference `audit_log` columns `event`, `actor`, `subject_id`. Actual schema (`001_init.sql:177-185`): `id, hatchery_id, user_id, action, payload, created_at`. Any story that writes audit_log with the assumed columns fails at runtime. (This also affects the `// TODO(audit-log)` guidance in `code-design.md` §5/§8 — the helper spec there must match the real columns.)

4. **`alert_severity` enum has no `'critical'`.** Enum is `('high','medium','low')` (`001_init.sql:121`). E4 AC#5 + Task 6 branch on `severity = 'critical'` — the Epic-K critical-warning fan-out silently never fires.

### SYS-2 — RLS policy mis-citation (BLOCKER class, A3 + B1 + B2 + B3)
Policies are NOT where the stories say:
- `customers` RLS → actual `002_rls.sql:59` (`customers_rw`), NOT `001_init.sql`
- `hatchery_brand` RLS → actual `006_line_integration.sql:84` (`hatchery_brand_rw`), NOT `002_rls.sql`
- `batch_buyers` RLS → actual `002_rls.sql:84` (this one stories cite correctly)

Fix: every story Dev Notes "RLS Tables" section must cite the migration that actually contains the `CREATE POLICY`, not the one that creates the table.

### SYS-3 — `line_user_id NOT NULL` omitted from INSERT patterns (WARN, D2/D3/E3/E4)
`line_outbound_events.line_user_id` is `text NOT NULL` (`006_line_integration.sql:47`). Stories that INSERT into `line_outbound_events` must populate `line_user_id` (resolved from the customer's bound `line_id`) AND filter out customers where `line_id IS NULL` before inserting. D3/E4 do this; D2/E3 don't say to. Add the guard to every enqueue path.

### SYS-4 — Phantom middleware path (BLOCKER class, K2/S4/S9)
Multiple stories reference `proxy.ts` or `lib/supabase/middleware.ts` as the Next.js middleware. **Neither the `proxy.ts` claim nor `lib/supabase/middleware.ts` resolves consistently** — auditors found `proxy.ts` exists and sets only `x-pathname` (S4), but `lib/supabase/middleware.ts` does NOT exist (S9, K2). Resolve the actual middleware location once, then fix every reference. S9's ESLint `getSession` allowlist targets the phantom path — it must point at the real cookie-refresh call site or the ban protects nothing.

### SYS-5 — No ESLint config exists at all (BLOCKER, S9)
S9 says "confirm `eslint.config.mjs` exists; if not migrate from `.eslintrc.json`." Neither exists. `next lint` runs with Next's built-in defaults only. S9 must be rewritten: "create `eslint.config.mjs` from scratch" — materially more work than the story implies.

## Per-epic BLOCKERs (with evidence + fix)

### Epic C — highest risk (3 BLOCKERs, one FIX-AUTO-introduced)

**C2 — impossible join.** Story computes mean-D30-per-batch via `customer_cycles.batch_id`. `customer_cycles` has no `batch_id` (`001_init.sql:66-75`: `customer_id PK, cycle_day, expected_harvest, d30, d60, restock_in, updated_at`). There is no FK path cycle→batch. **Fix needs a schema decision, not a doc edit:** either (a) add `batch_id` to `customer_cycles` in a migration, or (b) re-spec C2 to use `batch_buyers.d30` (the denormalised per-(batch,customer) D30 that *does* exist). Recommend (b) — least scope.

**C3 — FIX-AUTO-introduced BLOCKER.** Story says buyers come from `batch_distributions` with columns `quantity`, `sold_at`. Reality: table is `batch_buyers` (`001_init.sql:97`), columns `pl_purchased` (not `quantity`), no `sold_at`. Also references `customer_cycles.batch_id`/`started_at` (don't exist). **Fix: revert the FIX-AUTO rename — restore `batch_buyers`, use `pl_purchased`, drop `sold_at`, re-spec the D30 source to `batch_buyers.d30`.**

**C4 — four bugs.** `pcr_results.updated_at` doesn't exist (real: `created_at`) → idempotency check errors. `batch_certs` table doesn't exist (C4 must create it). ON CONFLICT targets a `(batch_id,customer_id)` index that doesn't exist on `line_outbound_events`. Migration numbered `014` should be per SYS-1. **Fix: mechanical — swap `updated_at`→`created_at`, make `batch_certs` an explicit CREATE, fix the ON CONFLICT to a real index, renumber.**

**C1 — WARN.** `pcr:write` RBAC action doesn't exist; C1 self-documents adding it. Sequence C1's rbac.ts edit before its gates compile.

## Recommended sequence before flipping the agent-dev switch

1. **PO/Architect**: decide C2 (recommend: use `batch_buyers.d30`), E4 (recommend: add `'critical'` to enum), H2 (recommend: story matches code). 3 quick calls.
2. **Assign SYS-1 migration numbers** (table above) — one decision.
3. **Mechanical fix batch** — like the original FIX-AUTO `/ultrawork`, but every table/column/migration/path reference is `grep`-verified against the live tree before the edit lands. This reverts the bad C3 rename and fixes all SYS-2/3/4 + per-story mechanical BLOCKERs.
4. **Rewrite S9** (ESLint from scratch) and **author `DSR-SPEC.md`** (umbrella repo) to unblock S7.
5. Re-run this executability audit (or a spot-check of the fixed BLOCKERs) → confirm zero BLOCKER.
6. **Then** flip the switch.

Until step 5 returns zero BLOCKER, autonomous agents will fail on ≥16 of 42 stories at execution time.

## Process recommendation

Add to `code-design.md` §16 (review checklist) and the BMAD story-authoring flow: **every concrete reference in a story (migration file, table, column, RPC, RBAC action, source path) must be grep-verified against the live codebase at authoring time, and re-verified by the executability audit before the story is marked ready-for-dev.** Alignment audits (`01`) and executability audits (this doc) are different gates — both are required. The FIX-AUTO incident proves an alignment-only pass can ship execution-breaking bugs.
