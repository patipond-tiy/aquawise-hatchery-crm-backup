# Story D1: See Farms by Restock Urgency

Status: ready-for-dev

## Story

As a Rep,
I want to see a triaged list of farms with cycles ending soon, grouped by urgency,
so that I work the most urgent ones first.

## Acceptance Criteria

1. `/restock` lists all customers where `restockIn != null`
2. Groups: now (≤0d), week (≤14d), month (≤45d), later — thresholds sourced from `hatcheries.restock_thresholds` jsonb
3. Each row shows restock-in days, last D30, batches count
4. Stat cards aggregate counts and estimated PL volume per urgency group
5. Group thresholds are configurable per nursery via Settings (new Settings UI + `updateRestockThresholds()` server action)

## Dev Notes

### Architecture Constraints
- All API calls via `lib/api/index.ts` facade — never Supabase directly from pages
- Server actions at `app/[locale]/(dashboard)/settings/actions.ts`; call `requireActiveSubscription()` from `lib/billing/guard.ts` before any write
- `'use client'` opt-in only; server components by default
- No `any` in app code; `as` casts allowed only in adapters

### API Facade Methods
No new facade method is required for reading thresholds — `getNursery()` already returns them. The `updateRestockThresholds()` server action calls Supabase directly (service-scope update on `nurseries`) consistent with the pattern used by `updateProfile()` in `settings/actions.ts`.

### RBAC
- Only `owner` role may update nursery-level settings. Check `can(role, 'settings:write')` via `lib/rbac.ts` before executing the update inside the server action, in addition to the billing guard.
- `counter_staff` may view the restock page (read path) but should not see the threshold edit form.
- **Exclude `lab_tech` and `auditor`** from page access — both are **—** in persona matrix (`01-personas.md`). Lab has no business need for restock data; Auditor is a compliance role focused on batches/PCR, not customer outreach. Gate the `/restock` route to `owner` and `counter_staff` only.

### RLS Tables
| Table | Operation | Migration |
|-------|-----------|-----------|
| `nurseries` | SELECT `restock_thresholds` (via `getNursery`) | `010_restock_thresholds.sql` |
| `nurseries` | UPDATE `restock_thresholds` (via server action) | `010_restock_thresholds.sql` |

The UPDATE must be performed with the user's session (not service-role) so RLS enforces nursery membership. The existing `nurseries` UPDATE policy (`owner_update`) in `002_rls.sql` / `011_rls_tighten.sql` already covers this.
