# Story B1: See All Customers at a Glance

Status: ready-for-dev

## Story

As a Rep,
I want to browse my customer list, search by farm or owner name, and filter by lifecycle status,
so that I can quickly find the nursery operator I need.

## Acceptance Criteria

1. `/customers` lists all customers belonging only to this nursery (RLS-enforced — no cross-tenant leakage)
2. Search filters by `farm`, `farmEn`, and `name` fields (case-insensitive, client-side debounce)
3. Tabs: All / Active (`cycleDay != null`) / Restock (`restockIn ≤ 14`) / Concern
4. Each card shows farm name, owner, status chips, batches count, LTV, last D30
5. Card click navigates to customer detail; 💬 icon opens the `sendLine` modal
6. Customers without a `customer_cycles` row appear in the list (not silently dropped)

## Dev Notes

### Domain Framing — Critical
"Customers" in Epic B are the **~20–40 nursery operators** (`โรงอนุบาล`) the nursery sells post-larvae (PL) to. They are NOT end farms. The customer list is a **relationship dashboard**, not a sales-pipeline CRM. The "farm" vocabulary in the schema is scaffolding from an earlier iteration; the exact UX framing is a hypothesis pending validation with P'Bunjong in 2027. Do not over-invest in pipeline-CRM metaphors.

### Implementation Status
- **B1.i is ✅** — left-join fix landed in commit `ddf6ea3`. `lib/api/supabase.ts:104–122` now left-joins `customer_cycles` so customers without a cycle row are returned.
- **B1.t is 🟡** — unit regression test exists (`tests/api/list-customers.test.ts`); the **cross-tenant RLS test is missing** (deferred to P0.3 in MATRIX).
- **B1.v is ❌** — live Supabase verification not yet done.
- This story's remaining work is **RLS test + live verification only**. No UI changes required.

### RBAC
- `can(role, 'customer:read')` — passes for `owner`, `counter_staff`, `lab_tech`, `auditor` (data permission)
- **Page access:** `owner`, `counter_staff`, `auditor` only. **Exclude `lab_tech`** — Lab is **—** in persona matrix (`01-personas.md`); Lab has no business need to browse the customer list. The data permission `customer:read` includes Lab for API-level access (e.g., batch detail buyer lookups), but the `/customers` page route should gate Lab out with a `role !== 'lab_tech'` check or equivalent redirect.
- No `customer:write` needed for list view

### RLS Tables

- `customer_cycles` — left-joined; verify RLS permits read for all four roles
