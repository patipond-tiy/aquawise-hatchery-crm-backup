# Story A3: Set Up the Hatchery Profile

Status: ready-for-dev

## Story

As an Owner,
I want to fill in my hatchery name, address, LINE OA, logo, and brand color,
so that customers see a professional, branded experience when they receive Flex messages or visit the public scorecard.

## Acceptance Criteria

1. Settings → Profile inputs are controlled with `useState` (not dead `defaultValue` props)
2. "บันทึก" calls `updateProfile()` server action; on success a toast appears; on error the toast shows the error message
3. Logo upload sends the file to Supabase Storage bucket `hatchery-logos`; validates MIME type (jpeg, png, webp, gif only; ≤ 2 MB); writes the public URL to `hatchery_brand.logo_url`
4. Display name (TH/EN) and brand color picker values persist to `hatcheries` + `hatchery_brand` tables on save
5. Saved brand fields (logo URL, color, display names) are available to Flex Message templates via `hatchery_brand` rows read by the bot worker

## Dev Notes

### Architecture Constraints
- All API calls via `lib/api/index.ts` facade — never Supabase directly from pages
- Server actions at `app/[locale]/(dashboard)/<page>/actions.ts`
- Storage helper is `'server-only'` (`lib/supabase/storage.ts` imports `'server-only'`) — never call it from a client component
- `'use client'` opt-in only; server components are the default
- No `any` in app code; `as` casts allowed only in adapters

### Files to Touch
The implementation (`.i`) is already complete (commit `56a9492`). The following files should be read during verification — no code changes are expected unless a gap is discovered.

| File | Role |
|------|------|
| `app/[locale]/(dashboard)/settings/page.tsx` | Settings page; profile inputs controlled with `useState` |
| `app/[locale]/(dashboard)/settings/actions.ts` | `updateProfile()` server action; upserts `hatcheries` + `hatchery_brand`; calls `uploadHatcheryLogo` |
| `lib/supabase/storage.ts` | `uploadHatcheryLogo(hatcheryId, file)`; MIME validation; Storage upload + public URL |
| `supabase/migrations/012_storage_logos.sql` | Creates `hatchery-logos` bucket (public, 2 MB) and tenant-scoped RLS policies |
| `tests/settings/profile.test.ts` | 3 Vitest cases covering MIME rejection, save without logo, save with logo |

### API Facade Methods
No new `lib/api/index.ts` facade methods are required for A3. Profile reads go through the existing `getHatchery()` and `getHatcheryBrand()` (or equivalent) facade methods. Profile writes use a dedicated server action to support `audit_log` and billing guard without changing the facade surface.

### RBAC
- `can(role, 'settings:write')` returns `true` only for `owner` (see `lib/rbac.ts`)
- The server action additionally gates logo upload to `role === 'owner'` explicitly: `if (logoFile && role === 'owner') { ... }`
- `counter_staff` can call `updateProfile()` for text fields if the action's permission model is relaxed in future, but they cannot change the logo
- `lab_tech` and `auditor` have no write access to settings

### RLS Tables
| Table | Operation | Migration | Who |
|-------|-----------|-----------|-----|
| `hatcheries` | UPDATE | `011_rls_tighten.sql` | `owner` only |
| `hatchery_brand` | UPSERT (INSERT + UPDATE) | `002_rls.sql` | hatchery members (all roles can read; brand update governed by app-layer role check) |
| `storage.objects` in `hatchery-logos` | SELECT | `012_storage_logos.sql` | public (no auth required) |
| `storage.objects` in `hatchery-logos` | INSERT, UPDATE, DELETE | `012_storage_logos.sql` | authenticated users in the same hatchery |

Note: `hatchery_brand` RLS intentionally allows all hatchery members to update brand assets day-to-day. The `owner`-only constraint on logo upload is enforced at the app layer (`role === 'owner'` check in the server action), not at RLS level.

### Storage Details
- Bucket name: `hatchery-logos`
- Public: `true` (logos are served without auth; required for Flex messages and the public scorecard)
- File size limit: 2 MB (`file_size_limit: 2097152`)
- Allowed MIME types: `image/jpeg`, `image/png`, `image/webp`, `image/gif`
- Path layout: `{hatchery_id}/logo.{ext}` — deterministic so re-uploads overwrite the previous file
- Upload mode: `upsert: true` — no duplicate objects accumulate
- Public URL: retrieved via `supabase.storage.from('hatchery-logos').getPublicUrl(path)`

### `ProfileFields` Interface
```ts
export interface ProfileFields {
  name: string;           // hatcheries.name
  name_en: string;        // hatcheries.name_en
  location: string;       // hatcheries.location
  location_en: string;    // hatcheries.location_en
  display_name_th: string; // hatchery_brand.display_name_th
  display_name_en: string; // hatchery_brand.display_name_en
  brand_color: string;    // hatchery_brand.brand_color
  logoFile?: File | null; // transient; not persisted directly
}
```

The action accepts `Omit<ProfileFields, 'logoFile'>` plus an optional `logoFile?: File | null` as a second parameter so the file can be passed from a `<form>` action or called directly from a button handler.
