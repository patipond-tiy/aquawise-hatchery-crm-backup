# Story A6: Social Sign-In — Google + LINE

Status: done

## Story

As an Owner signing up or signing in,
I want to authenticate with Google or LINE in one tap (with the existing email
magic-link kept as a fallback),
so that I can get into the CRM without waiting for an email, using the identity
I already carry on my phone.

> **Scope origin:** Not in the original A-epic backlog. A1 is explicitly
> "magic-link, no password". This story is a deliberate, user-requested scope
> addition that *extends* A1's entry path; it does not rewrite A1. A4 (sign-out)
> is already provider-agnostic (`supabase.auth.signOut()`) and is **not**
> changed — only re-verified to cover social sessions (AC #9).

## Acceptance Criteria

1. `/{locale}/login` shows three ways in, in this order: **Continue with Google**,
   **Continue with LINE**, then the existing **email magic-link** form below a
   divider ("หรือ" / "or"). Magic-link behaviour is byte-unchanged from A1.
2. **Google:** clicking *Continue with Google* calls
   `supabase.auth.signInWithOAuth({ provider: 'google', options: { redirectTo: ${origin}/auth/callback } })`.
   The existing `app/auth/callback/page.tsx` PKCE `?code=` branch
   (`exchangeCodeAction` → `bootstrapNursery`) handles the return with **no
   changes**. First-ever Google sign-in for an email bootstraps a workspace
   exactly as A1 (one `nurseries` + one `owner` membership + 30-day trial).
3. **LINE:** clicking *Continue with LINE* navigates to `/auth/line/start`
   (a Route Handler, not a page). It generates a CSPRNG `state` + `nonce`,
   stores them in `httpOnly; Secure; SameSite=Lax`, short-TTL cookies, and
   302-redirects to `https://access.line.me/oauth2/v2.1/authorize`
   (`response_type=code`, `scope=openid profile email`, our LINE **Login**
   channel id, `redirect_uri=${origin}/auth/line/callback`, `state`, `nonce`).
4. **LINE callback** `/auth/line/callback` (Route Handler): verifies `state`
   matches the cookie (CSRF); exchanges the code at
   `https://api.line.me/oauth2/v2.1/token`; verifies the returned `id_token`
   at `https://api.line.me/oauth2/v2.1/verify` asserting
   `iss=https://access.line.me`, `aud=<LINE_LOGIN_CHANNEL_ID>`, not expired,
   and `nonce` equals the cookie. Any failure → redirect to
   `/{locale}/login?error=line_failed` with **no token/exception text in the URL**.
5. **LINE → Supabase bridge (no implicit fragment):** using
   `createServiceClient()` inside the callback Route Handler only, resolve the
   Supabase user: look up by the LINE-asserted email if present, else by the
   `line_identities` row keyed on the LINE `sub`. Create the auth user if none
   (`auth.admin.createUser`, `email_confirm: true`). Mint the session
   **server-side**: `auth.admin.generateLink({ type: 'magiclink' })` →
   `auth.verifyOtp({ type: 'magiclink', token_hash })` on the SSR client so the
   session cookie is set directly in the handler. **The browser is never sent
   an `access_token`/`refresh_token` URL fragment** (closes security.md §3/§4
   implicit-flow + open-redirect exposure). Then `bootstrapNursery(user.id)` and
   302 to `/{locale}`.
6. A `line_identities` table (migration **027**) maps `line_sub` (unique) →
   `auth.users.id`, with `email_at_link`, `display_name`, `created_at`. RLS
   enabled; a row is readable only by its own `user_id`; all writes are
   service-role only (no end-user INSERT/UPDATE/DELETE policy). Added to
   `supabase/tests/cross-tenant.sql`.
7. **LINE email may be absent** (LINE only returns it when the channel has the
   email permission *and* the user consents). When absent, the user is keyed
   solely by `line_identities.line_sub`; `auth.users.email` is set to a
   reserved synthetic `line_${sub}@line.nursery.local` that is **never shown
   in UI** and **never used to send mail**. Documented limitation: a LINE-only
   (no-email) account does **not** auto-merge with a later magic-link/Google
   account for the same human — explicitly recorded in Dev Notes + provisioning
   doc as known v1 behaviour, not a defect.
8. Mock mode / unconfigured: if `isMockMode()` or `LINE_LOGIN_CHANNEL_ID` is
   unset, the LINE button is still rendered but `/auth/line/start` redirects to
   `/{locale}/login?error=line_unconfigured` and the page shows a friendly Thai
   inline notice. Google button hidden when `isMockMode()` (no Supabase to
   round-trip). Magic-link path unaffected in all modes.
9. **Sign-out (A4) re-verified, not modified:** signing in via Google or LINE
   then clicking "ออกจากระบบ" clears the session and `queryClient.clear()` and
   redirects to `/{locale}/login` exactly as A4; navigating back to a dashboard
   route redirects to login (no stale social session). No A4 code change.
10. All new user-facing strings exist in **both** `messages/en.json` and
    `messages/th.json` under a new `auth`/`login` namespace; Thai is natural
    (Thai-first product). No emoji on the login surface (conformance §1).

### Architecture Constraints
- All API calls via `lib/api` facade — but auth-path logic lives in Route
  Handlers / `app/auth/**`, mirroring A1 (`app/auth/callback/actions.ts`),
  **not** the facade. Bridge uses `createServiceClient()` from
  `lib/supabase/server.ts` — confined to the callback Route Handler.
- `getUser()` not `getSession()` for any auth decision (security.md §14).
- No `any`; `as` only in the LINE token/verify response adapters.
- Login page is already `'use client'` (CD-1 tracked debt) — do not add new
  `'use client'` pages; the two LINE endpoints are Route Handlers (no UI).

### Security (security.md alignment)
- **§3 implicit-flow / §4 open-redirect:** the LINE bridge mints the session
  **server-side** via `generateLink` → `verifyOtp` and sets cookies in the
  handler. No `access_token` in a URL fragment, no client-controlled
  `redirect_to` — the post-login destination is the fixed `/{locale}`.
- **CSRF:** `state` cookie compared constant-time to the callback `state`.
- **Replay:** `nonce` echoed into LINE `id_token` and asserted on verify.
- **Token leakage:** id_token / access_token / `line_sub` never logged, never
  placed in a redirect URL; failure → opaque `?error=line_failed`.
- **Secrets:** `LINE_LOGIN_CHANNEL_SECRET` is server-only (no `NEXT_PUBLIC_`);
  service-role key stays server-side (Route Handler runtime).
- **D-007:** this introduces a 4th sanctioned `createServiceClient()` site
  (auth-provider callback) beyond "Stripe webhook / LINE worker / cron".
  Recorded in DECISIONS.md D-007 and conformance-gate §3; security.md §14
  mitigation list updated to name this site explicitly.
