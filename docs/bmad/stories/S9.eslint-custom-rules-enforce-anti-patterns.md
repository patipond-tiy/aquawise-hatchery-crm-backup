# Story S9: ESLint Custom Rules to Enforce §18 Anti-Patterns

Status: ready-for-dev

Severity: P1 — pre-launch (shifts §18 enforcement left, from review-time to lint-time)

## Story

As a Developer,
I want three §18 anti-patterns to fail `pnpm lint`,
so that the reviewer never has to mention them and the codebase never quietly accumulates them.

## Acceptance Criteria

1. `pnpm lint` fails when:
   - `getSession` is called outside `lib/supabase/middleware.ts` (the only file where the rule is suppressed; note: `middleware.ts` itself uses `getUser()` for cookie refresh — the allowlist exists to avoid blanket-suppressing the file in future, not because it currently calls `getSession`)
   - `dangerouslySetInnerHTML` is used anywhere in `app/` or `components/`
   - A role-string equality comparison is used (`role === 'owner'`, `member.role === 'counter_staff'`, etc.) outside `lib/rbac.ts`
2. The current codebase passes the new rules (relies on S6 having landed first)
3. Each rule produces a clear error message pointing the developer at the §18 anti-pattern it enforces and at the `can()`/server-only alternative
4. Rules are configured via `eslint.config.mjs` (flat config) — not `.eslintrc.json`
5. `next lint` continues to work with the same config

## Tasks / Subtasks

  - [ ] Role-string ban: same — `no-restricted-syntax` with a selector like `BinaryExpression[operator='==='][left.property.name='role'][right.value=/owner|counter_staff|lab_tech|auditor/]`

## Dev Notes

### Architecture Constraints
- Flat config is the only supported format in ESLint 9. This project has no ESLint config at all — Task 2 creates `eslint.config.mjs` from scratch (not a migration)
- AST selectors are syntactic, not semantic — there will be false positives (e.g., `someOtherObject.role === 'owner'` would trip the rule). Tighten the selector pattern if false-positive count is high after Task 5
