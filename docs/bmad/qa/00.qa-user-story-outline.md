# QA Outline — User Story ↔ Business Plan Alignment Audit (template)

> **What this file is.** A reusable spec for running a story-alignment QA audit on any project. Copy it, swap the `{{PLACEHOLDERS}}` for your project's specifics, and you have a ready-to-fire QA brief. Pair it with the workflow at the bottom (`deep-interview → spec → ultrawork → assembly`) to produce the actual audit doc.
>
> Sections below mirror the same structure used to scope the AquaWise audit (`00.qa-user-story.md`). Edit each section; do not delete the structure.

---

## Title

`{{PROJECT_NAME}} Story ↔ Business Plan Alignment Audit`

→ Example: `BMad Story ↔ Business Plan Alignment Audit`

---

## Purpose

Surface specific `{{PROJECT_NAME}}` user stories that drift from the `{{PROJECT_NAME}}` business plan / project vision, so each can be triaged (`KEEP` / `FIX` / `CUT` / `DEFER`) before further dev effort goes into them.

→ Adapt the verdict set to your team's lifecycle (e.g., add `BLOCK` if you have a release gate, drop `DEFER` if you don't version).

---

## Output

One markdown file (proposed location `{{OUTPUT_PATH}}`, e.g., `docs/qa/story-alignment-audit.md`), containing a single scorecard table plus a summary section at the top.

→ Keep it in **one file**. A multi-file QA audit is unscannable.

---

## Scope

All `{{N}}` stories under `{{STORIES_PATH}}` regardless of status (`done`, `ready-for-dev`, `draft`, …).

→ Default: audit everything. Filter by status only if your story count is unwieldy (>200) or the audit is recurring (then audit deltas only).

---

## Reference Spec (what we audit *against*)

Primary reference: `{{BUSINESS_PLAN_PATH}}` — specifically the sections:

- `{{PRINCIPLE_1}}` — *the project's master decision filter (one-sentence question)*
- `{{PRINCIPLE_2}}` — *the phased roadmap or evolution model (if any)*
- `{{PRINCIPLE_3}}` — *the non-negotiable rules / red lines*
- `{{PRINCIPLE_4}}` — *the emotional / user-truth promise*

→ Examples (from AquaWise): "100-farm filter", "The Three Phases (advisor → personalized → cross-farm)", "The Trust Rules (5 non-negotiables)", "The Emotional Truth ('less alone, less cheated, less blind')".

→ If your project doesn't have all four, drop the missing one from the table columns. If you have a 5th (e.g., a regulatory checklist), add it as a column. Keep the structure symmetric — one column per audited principle.

---

## The Audit Lenses (default set: 5 lenses)

| # | Lens | What it checks | Adaptation note |
|---|------|----------------|-----------------|
| 1 | **`{{MASTER_FILTER}}`** | Does this story measurably move us toward the project's primary success measure? | The most important lens — should be a single yes/no question from your business plan |
| 2 | **`{{PHASE_FIT}}`** | Which phase of the roadmap does this serve? Does it ship later-phase capability before earlier-phase prerequisites are proven? | If you don't have phases, drop this column |
| 3 | **`{{NON_NEGOTIABLES}}`** | Does the story respect each non-negotiable rule? (Cite specific rule # + reason) | Number your rules so cells can reference `rule#X` |
| 4 | **`{{EMOTIONAL_PROMISE}}`** | Does the story serve the user's emotional truth, or does it feel like the opposite (surveillance, gating, condescension, etc.)? | Drop or rename if your product is purely B2B / infrastructure |
| 5 | **Internal clarity** | Are the story file's own AC / user story / dev-notes unambiguous and testable? *(Story-quality check, not alignment.)* | Always keep this one — it catches story-quality drift independent of vision |

→ The default 5 lenses come from the AquaWise audit. Swap, add, or drop based on which lenses you'd actually act on if a story failed them. If you wouldn't take action, don't add the column.

## Table Shape

```
┌───────┬───────────┬───────┬───────────────┬──────────────┬──────────┬─────────┐
│ Story │ {{Lens1}} │ {{L2}}│ {{Lens3}}     │ {{Lens4}}    │ Clarity  │ Action  │
├───────┼───────────┼───────┼───────────────┼──────────────┼──────────┼─────────┤
│ <id>  │ <verdict> │ <ph>  │ <rule+reason> │ <e-promise>  │ <ambig>  │ <verb>  │
└───────┴───────────┴───────┴───────────────┴──────────────┴──────────┴─────────┘
```

Columns = `Story` + one column per lens + `Action`. Default: 7 columns.

The `Story` cell should be a markdown link to the story file so readers can drill down.

---

## Cell Conventions

For lenses 1–5:

- `✓` — clean / aligned
- `⚠️ <reason>` — concern; cite the specific AC# or short quote from the story file
- `❌ <reason>` — violation; cite the specific AC# or short quote
- `neutral` — story is enabling/infra (e.g., a migration); the lens doesn't directly apply
- For the Phase column specifically: name the phase served (`P1` / `P2` / `P3` / `P1→P2` / etc.)

→ The `⚠️`-must-cite-evidence rule is **the most important convention**. Without it, you get vibes-based audits that nobody can verify.

---

## Action Verdicts

The final column carries exactly one of:

- `KEEP` — ship as-is
- `FIX: <one-line direction>` — keep the story but the noted concern must be addressed first
- `CUT` — drop from the current scope; misaligned with vision
- `DEFER to {{NEXT_VERSION}}` — premature; move to a later phase when prerequisites are met

→ The action column is the only output your stakeholders actually read. Do not allow `TBD` or `review later` — every row must have a decision.

---

## Acceptance Criteria (for the audit document itself)

- [ ] All `{{N}}` stories appear as rows; none missing
- [ ] Every cell is filled — no blanks, no `TBD`. Use `neutral` with a reason where a lens genuinely doesn't apply
- [ ] Every `⚠️` and `❌` cell quotes a specific AC# or short phrase from the story file as evidence
- [ ] Action column has a verdict for every row
- [ ] **Summary block at top** with: (a) counts of `KEEP` / `FIX` / `CUT` / `DEFER`, (b) the 3–5 highest-priority `FIX`es (biased toward non-negotiables and the master filter), (c) one paragraph on notable drift patterns observed across all stories
- [ ] A reader unfamiliar with the codebase can scan the doc in **<10 min** and know what to fix next

---

## Non-Goals (explicitly excluded)

- Not editing the story files themselves — the audit is read-only on source stories
- Not auditing higher-level docs (epics, PRDs, architecture) — only the per-story files
- Not building a reusable scoring rubric — this is a **one-shot audit** at a point in time, not a framework
- Not making the fix-or-cut decision unilaterally — the doc *recommends*, a human ratifies

→ Adapt these non-goals if your audit serves a different purpose (e.g., if you DO want it to be a recurring framework, remove non-goal #3 and add a "re-run cadence" section).

## How to Fill This In (workflow)

Recommended 4-step workflow:

1. **Deep-interview** the audit owner against this outline to nail down all `{{PLACEHOLDERS}}`. Use `/deep-interview` (or any structured Q&A) until ambiguity is below 20% — the lenses, scope, and reference doc must be unambiguous before any judging starts.
2. **Crystallize the spec** as a new file (e.g., `docs/qa/{{audit-slug}}.md`) containing the filled-in version of this outline plus an empty scorecard table with one row per story.
3. **Run the audit** via parallel workers (recommended: `/ultrawork`). Split your N stories into chunks (typically by epic letter or feature area). Each worker reads only its assigned story files plus the reference spec, returns markdown rows.
4. **Assemble** — one main agent merges the worker outputs into the table, then writes the Summary block (counts + top fixes + drift patterns).

## Generic `/ultrawork` Command Template

Replace `{{...}}` placeholders, then paste into a Claude Code prompt:

````
/ultrawork Audit all {{N}} user stories listed in {{AUDIT_DOC_PATH}} against the {{PROJECT_NAME}} business plan. Read this file first to understand the {{NUM_LENSES}} lenses, cell conventions, action verdicts, and acceptance criteria — then read {{BUSINESS_PLAN_PATH}} as the reference spec, then read each of the {{N}} story files in {{STORIES_PATH}} (skip {{AUDIT_DOC_PATH}} itself).

For each story, fill the row in the Scorecard table with these {{NUM_COLUMNS}} cell values:
  1. {{LENS_1}}: ✓ / ⚠️ <reason> / ❌ <reason> / neutral
  2. {{LENS_2}}: <phase or P1/P2/P3-style classification>
  3. {{LENS_3}}: ✓ / ⚠️ <rule# + reason citing AC#> / ❌ <rule# + reason citing AC#>
  4. {{LENS_4}}: ✓ / ⚠️ <reason> / ❌ <reason> / neutral
  5. Clarity: ✓ / ⚠️ <which AC# + why ambiguous> / ❌ <which AC# + why ambiguous>
  6. Action: KEEP / FIX: <one-line direction> / CUT / DEFER to {{NEXT_VERSION}}

Hard rules:
- Every ⚠️ and ❌ MUST quote a specific AC number or short phrase from the story file. No vague verdicts.
- No "TBD" or blank cells. Use "neutral" with a reason if a lens doesn't apply.
- Action column must be decisive.
- Do not edit any story file. Only edit {{AUDIT_DOC_PATH}}.

After all {{N}} rows are filled, update the Summary block at the top with:
  - Counts for KEEP / FIX / CUT / DEFER
  - The top 3–5 highest-priority FIXes (bias toward those protecting {{MASTER_FILTER}}, {{NON_NEGOTIABLES}}, or phase discipline)
  - One paragraph on notable drift patterns observed

Parallelization: split the {{N}} stories into {{NUM_CHUNKS}} chunks by {{CHUNK_KEY, e.g., "epic letter" or "feature area"}} — one worker per chunk. Each worker reads only its assigned story files plus the reference docs. Workers must NOT modify each other's table rows; coordinate via line-anchored edits (each row uniquely identified by its `[Story]` link).

When all workers finish, one final worker writes the Summary block based on the populated table.
````

---

## Notes for the audit owner

- **Time budget.** A 70-story audit with 8 parallel workers takes ~5–15 min of wall time and produces a ~50 KB doc. Scale roughly linearly with story count if you hold workers constant.
- **Worker-edit conflict risk.** If two workers try to write the same file simultaneously, one may lose its edits. The safer pattern (used in the AquaWise audit) is: workers RETURN markdown rows as text, the orchestrator does one consolidated `Edit` of the full table body. Adjust the ultrawork prompt if you hit this.
- **Reading the result.** The Summary block is the executive read; the table is the engineering drill-down. If your stakeholders skip the Summary, the audit has failed its scannability AC.
- **Cell length.** Evidence citations make cells verbose. That's the trade-off — verifiable verdicts beat scannable-but-vibes verdicts. If readability suffers badly, add a `Severity` summary column (`P0`/`P1`/`P2`) so the Summary block can sort by it.
- **What this outline is NOT.** It is not a perpetual QA framework. It is the *template* for running a one-shot point-in-time audit. Re-run the audit per release, not per commit.
