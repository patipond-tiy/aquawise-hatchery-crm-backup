# Story K6: Add `batches.species` Column

Status: ready-for-dev

> Cross-product gap closed. Conforms to `docs/aquawise-updated-docs/K-INTEGRATION-CONTRACT.md` §5/§8. Required by line-bot K13 (repurchase-prompt threshold logic needs species; line-bot `decisions-2026-05-15-fix-review.md` ~line 227 recommends *"opening a hatchery-CRM follow-up story to add `batches.species` (default 'vannamei')"*). Without it, line-bot K13 falls back to `line_users.species_preference` and accepts whiteleg-vs-tiger approximation.

## Story

As the LINE bot's K13 repurchase-prompt logic,
I want the batch's species returned by the CRM batch-read API,
so that harvest-window and dosage thresholds use the hatchery-supplied species (P. vannamei vs P. monodon) instead of an approximate per-user preference.

## Acceptance Criteria

1. **Migration `026_batches_species.sql`** (SYS-1 ledger: 013–025 reserved; K6 = **026**): `ALTER TABLE public.batches ADD COLUMN species text NOT NULL DEFAULT 'vannamei' CHECK (species IN ('vannamei','monodon'))`. Back-fills existing rows to `'vannamei'` via the DEFAULT during the ALTER (the dominant Thai species; correct for the existing seed/prod set).
2. **Regenerate types:** `pnpm supabase gen types typescript --linked > lib/database.types.ts` so `Database['public']['Tables']['batches']['Row']` carries `species`.
3. **K2 read payload** (`GET /api/v1/batches/:code`) includes `"species": "vannamei" | "monodon"` per K-INTEGRATION-CONTRACT §5. (This story owns adding the field to the K2 serializer; coordinate the one-line serializer edit with whoever implements K2 — see Dependencies.)
4. **Add-batch surface:** the batch-register flow (Story C1 / the Add Batch modal) gets a species selector (`vannamei` default-selected, `monodon` alternative). Two i18n keys only; default preserves current behavior so C1 is not blocked by K6.
5. **No RLS/RBAC change:** `species` is a non-sensitive attribute on an already-RLS-scoped table; existing `batches` policies cover it.

## Dev Notes

### Architecture Constraints
- Migration follows the existing template (RLS already enabled on `batches` from `001_init.sql` / `002_rls.sql`; ALTER inherits it — no new policy).
- No `any`; `as` only in the K2 row mapper (architecture.md §2 rule 3).
- The `CHECK` constraint is the source of truth for allowed species; the UI selector and the contract enum must match it. If a third species is ever needed, it is a migration + contract change, not a UI-only change.
