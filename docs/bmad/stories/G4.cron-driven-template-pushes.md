# Story G4: Cron-Driven Template Pushes

Status: draft

## Story

As an Owner,
I want restock reminders and harvest-window alerts pushed automatically to farmers without anyone clicking anything,
so that LINE outreach is consistent and no restock opportunity is missed.

## Acceptance Criteria

1. A daily cron fires at 09:00 ICT via Vercel cron (`vercel.json`) and calls a protected API route `app/api/cron/daily/route.ts`
2. The cron evaluates all `customers` where `restock_in ∈ {7, 3, 0}` days across all hatcheries and enqueues `restock_reminder` events into `line_outbound_events`
3. The cron evaluates upcoming harvest windows and enqueues `harvest_window` events
4. Idempotency is enforced via the existing partial unique index on `(customer_id, template, payload->>'cycle_id')` — running the cron twice in one day inserts no duplicate rows
5. Per-hatchery notification settings are respected: if `notification_settings.restock = false` for a hatchery, no events are enqueued for that hatchery's customers on that run
6. The route is protected by `CRON_SECRET` — requests without the correct `Authorization: Bearer <CRON_SECRET>` header return 401

## Dev Notes

### Architecture Constraints

- Cron route is a Next.js API Route (`app/api/cron/daily/route.ts`), not a Server Action — Vercel cron calls HTTP endpoints, not server actions
- Use service-role Supabase client inside the cron route (no session cookie available in cron context)
- All pushes go through `line_outbound_events` — cron never calls LINE Messaging API directly
- No `any` in app code; narrow types in the cron helpers

### Blockers

**G4 is blocked on:**
- **G3p.i** — the bot worker must exist and be consuming `line_outbound_events` before cron-enqueued events can be delivered. The cron route itself can be written and tested independently, but `.v` cannot pass until G3p ships.
- **H1.i** (delivery-time check) — `notification_settings` toggles must actually suppress enqueue. H1 toggles persist to DB today; the cron must read them before inserting. These are in the same cron route, so this is an implementation detail of Task 1 above, not a separate blocker.

### Idempotency Detail

The partial unique index in `006_line_integration.sql` handles deduplication:

```sql
UNIQUE (customer_id, template, (payload->>'cycle_id'))
  WHERE status IN ('pending', 'sending', 'sent')
  AND template IN ('restock_reminder', 'harvest_window')
```

Insert pattern for the cron (use a Supabase RPC or raw SQL to leverage the ON CONFLICT):

```sql
INSERT INTO line_outbound_events (hatchery_id, customer_id, template, payload, status, kind)
SELECT ..., 'restock_reminder', $payload::jsonb, 'pending', 'template_push'
FROM qualifying_customers
ON CONFLICT (customer_id, template, (payload->>'cycle_id'))
  WHERE status IN ('pending','sending','sent')
  AND template IN ('restock_reminder','harvest_window')
DO NOTHING;
```

The Supabase JS client `.upsert()` cannot express this partial index directly — use `supabase.rpc('enqueue_restock_events', {...})` or a raw `supabase.from(...).insert(rows, { ignoreDuplicates: true })` only if the conflict target is unambiguous.

### Threshold Configuration

Restock thresholds (`now: 0d`, `week: 7d`, `3d`) come from `hatcheries.restock_thresholds jsonb` (added in migration 010). The cron should evaluate against the hatchery's own configured thresholds, not hardcoded values. Default: `{7, 3, 0}` (days before restock).

### RBAC

- Cron route uses service-role — no user session, no RBAC check
- The route is protected by `CRON_SECRET` header only; Vercel injects this automatically on production crons

### Vercel Cron Notes

- Vercel cron is available on Pro plan and above; the cron expression `0 2 * * *` fires once daily at 02:00 UTC (09:00 ICT)
- Vercel automatically sets `Authorization: Bearer <VERCEL_CRON_SECRET>` on cron invocations — set `CRON_SECRET` in Vercel env vars to the same value
- Cron timeout: Vercel enforces a 60s max execution per cron invocation on Pro; paginate hatchery queries if the hatchery count grows
