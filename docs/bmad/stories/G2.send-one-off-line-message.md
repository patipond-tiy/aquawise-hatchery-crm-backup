# Story G2: Send a One-Off LINE Message to a Single Customer

Status: draft

## Story

As a Rep,
I want to open a customer and tap "‡∏™‡πà‡∏á LINE" with a template (cycle-end reminder / new lot / promo / custom note),
so that I message them in two clicks without leaving the CRM.

## Acceptance Criteria

1. `sendLine` modal on the customer card/detail lists available templates and a free-text option
2. Rep selects a template (or writes a custom note), fills in any required fields, and submits
3. Submit calls the `sendLineEvent(customerId, template, payload)` server action which inserts one row into `line_outbound_events` with `status='pending'`
4. An optimistic toast appears in the CRM immediately; the customer Activity panel shows the event as "pending"
5. Bot worker (in `aquawise-line-bot`) consumes the queue, renders the Flex with `hatchery_brand`, pushes via @aquawise OA
6. Status updates from `pending ‚Üí sending ‚Üí sent` (or `failed ‚Üí dead`) are reflected in the customer Activity panel
7. The send is not blocked by quiet hours (rep is making an intentional human decision); cron-driven sends respect quiet hours

## Dev Notes

### Architecture Constraints

- All API calls via `lib/api/index.ts` facade ‚Äî never Supabase directly from pages
- Server actions at `app/[locale]/(dashboard)/customers/[id]/actions.ts`
- `'use client'` opt-in only; the modal component is already client-side; server action is called via `useTransition`
- No `any` in app code; `as` casts allowed only in adapters
- CRM never calls LINE Messaging API directly ‚Äî all pushes go through `line_outbound_events`

### Blockers

**G2 is blocked on:**
- **G3p.i** ‚Äî the bot worker consumer must exist and be running before any `line_outbound_events` row will be delivered. G2's server action can be written and tested independently, but `.v` cannot pass until G3p ships.
- **G1.i** ‚Äî a customer must have a bound `line_id` for the bot worker to push to them. Write/test G2 against a mock customer; live verification requires G1 to be complete.

### Template Payload Contract

The CRM is responsible for constructing the `payload` object before inserting; the bot worker renders from it. The payload shape for templates available to manual send:

```jsonc
// restock_reminder
{
  "hatchery_id": "uuid",
  "customer_id": "uuid",
  "cycle_id": "uuid",           // current active cycle
  "days_until_restock": 7,
  "last_d30": 82,
  "preferred_size": 12
}

// new_batch_announcement ‚Äî payload constructed by rep
{
  "hatchery_id": "uuid",
  "customer_id": "uuid",
  "batch_id": "B-2604-A",
  "available_qty": 500000,
  "pcr_status": "clean"
}

// custom_note ‚Äî free text maps to a simple template
{
  "hatchery_id": "uuid",
  "customer_id": "uuid",
  "note": "string (max 300 chars)"
}
```

### RBAC

- `counter_staff`, `owner` can send LINE events (`customer:write`)
- `lab_tech` and `auditor` cannot
- Use `can(role, 'customer:write')` from `lib/rbac.ts`

### Billing Gate

`sendLineEvent()` must call `requireActiveSubscription()` before inserting. When `trial_expired` or `canceled`, the server action throws a `PaywallError` (HTTP 402) and the modal shows an inline "Subscribe to continue" prompt ‚Äî not a redirect.
