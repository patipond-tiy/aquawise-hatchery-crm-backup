# Story S4: Content-Security-Policy Header with Nonce + strict-dynamic

Status: ready-for-dev

Severity: P1 — pre-launch hardening

## Story

As an Owner,
I want a strict Content-Security-Policy header on every response,
so that even if an XSS bug slips past React's escaping, the attacker cannot exfiltrate data to a third-party domain or load attacker-controlled scripts.

## Acceptance Criteria

1. `proxy.ts` generates a fresh nonce per request and sets a CSP header that uses `'nonce-<value>' 'strict-dynamic'` for `script-src`
2. The CSP allows: self, Supabase, Stripe (incl. js.stripe.com iframe), Sentry (when added), LINE API
3. The CSP **disallows**: inline scripts without nonce, `unsafe-eval`, `object-src`, `frame-ancestors` (other than `'self'`)
4. `X-Content-Type-Options: nosniff`, `Referrer-Policy: strict-origin-when-cross-origin`, and `Permissions-Policy` (camera/mic/geo denied) are also set
5. Strict-Transport-Security is set via `next.config.mjs` `headers()` for production only
6. Mock-mode click-through of the full dashboard works without CSP violations in DevTools console
7. Stripe Checkout iframe loads and completes a test payment

## Dev Notes

### Architecture Constraints
- `proxy.ts` is Next.js 16's rename of `middleware.ts` — already exists, runs `next-intl` middleware AND injects `x-pathname`. Add nonce + CSP to the same response object.
- Nonces require dynamic rendering — incompatible with PPR. The decision-log entry documents this.
- Tailwind 4 uses CSS-first tokens (no JS); no nonce needed for styles.
